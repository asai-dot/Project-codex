# 案件 / 非案件の切り分けロジック ― トリアージ設計

- 作成日: 2026-06-05
- 用途: 久保さんへの設計材料。紐付けの**手前にある最難関の論点**「これは案件か、非案件か」をどう捌くか。
- 関連: `02_案件紐付けロジック`（どの案件か）／`03_紐付け精度を上げる要素`（何を手掛かりに）。本書は**その前段のゲート**。

---

## 1. なぜ独立した論点なのか ― 2つの失敗モード

入ってくるもの（メール/FAX/郵便/電話/打合せ）を、まず **案件（どこかの受任案件・相談に属す）／非案件（研修・所内・ベンダー・採用・通知・メルマガ等）** に仕分ける必要がある。ここを誤ると2方向に壊れる。

| 失敗モード | 症状 | コスト |
|---|---|---|
| **過剰探索**（延々と案件を探す） | 非案件なのに、合う案件を探し続ける／無理やり当てる | 計算資源の浪費＋**誤紐付けのノイズ**を量産 |
| **紐付け漏れ**（取りこぼし） | 本物の案件メール・郵便を未分類のまま放置 | **最悪**。期日通知・依頼者連絡の見落とし＝事故リスク |

**この2つは相反する**。漏れを恐れて何でも案件候補にすると過剰探索になり、過剰探索を嫌って早く切ると漏れる。**両立させる設計が本書の主題。**

---

## 2. 設計の大原則 ― コストの非対称性

> 法律事務所では **「紐付け漏れ ＞ 過剰取り込み」**。漏れは事故に直結する。だから迷ったら**捨てずに人に回す**。ただし「延々探す」は別問題として、探索コストに上限を設ける。

この原則を、相反しない形に分解する。

1. **疑わしきは捨てない（recall優先）** ― 判定不能は「未分類トレイ」へ積み、**人が必ず見る**。漏れの最後の砦。
2. **でも延々は探さない（コスト上限）** ― 探索は有限手で打ち切る。打ち切り＝**諦め**ではなく**「未分類トレイへ送る」**こと。
3. **非案件は『探さない』のではなく『先に非案件と確定させる』** ― 積極的に非案件と判定できれば、探索をそもそも始めない。これが過剰探索を止める最大の鍵。

→ つまり「**非案件を積極判定して早く外す**（precision）」＋「**判定できないものは未分類で必ず人へ**（recall）」の二段構え。

---

## 3. 2ゲート構造（探索を止める仕組み）

紐付けを1段でやると「合う案件が見つからない＝延々探す」になりがち。**案件性ゲートを先に置く。**

```
入ってくるもの
   │
   ▼
┌─────────────────────┐
│ ゲート1: 案件性ゲート（これは案件か？）   │  ← 案件らしさ/非案件らしさをスコア
└─────────────────────┘
   │
   ├─ 非案件と積極確定 ──▶ 非案件バケットへ（探索しない）
   │
   ├─ 案件確度が高い ────▶ ゲート2へ
   │
   └─ グレー（判定不能）──▶ 未分類トレイ（人レビュー）
                              │
   ┌─────────────────────┐
   ▼                                            │
┌─────────────────────┐         │
│ ゲート2: 紐付けゲート（どの案件か？）     │ ←─┘（人が「案件」と判断したものも合流）
│  ＝ 02/03 のロジック・7段階ティア        │
└─────────────────────┘
   │
   ├─ 確信度 高 ──▶ 自動紐付け
   ├─ 確信度 中 ──▶ 「Salesforceのこれですか？」候補提示
   └─ 確信度 低 ──▶ 未分類トレイ（人レビュー）
```

**ポイント**: ゲート2で「候補スコアが閾値未満／候補ゼロ」になったら、**N手で打ち切って未分類トレイへ**。ここで諦めずに探し続けるのが「延々探す」の正体なので、**打ち切り条件を明示的に持つ**。

## 4. 案件性スコア ― 何で「案件らしさ」を測るか

ゲート1で使うシグナル。`03`の系統（主体/コンテンツ/流れ/活発度/内部受け手）を**案件性の判定にも流用**する。

### 案件らしさ ↑（加点）
- 既知の**依頼者・相手方・代理人**が当事者（主体系）
- **事件番号・裁判所名**が本文/書面にある（コンテンツ系）／**郵便の送達**（PostalMatter）
- **弁護士個人宛**で来ている（内部受け手）
- 既に**Box案件フォルダ／Gmail案件ラベル**が存在する依頼者
- 相談→受任の文脈、見積・委任契約のやり取り

### 非案件らしさ ↑（減点・早期確定）
- **一斉配信ヘッダ**（`List-Unsubscribe` 等）、no-reply、メルマガ
- **所内・運営系アドレス**: `info@` `recruite@`（採用）、`box@`（ストレージ通知）、Salesforce連携通知
- **研修・セミナー定型語**（〜研修/〜セミナー/講師依頼/受講案内）
- **ベンダー請求・事務用品・回線等**、自事務所の運営コスト
- **所内連絡のみ**（外部当事者がいない）

> 実データの裏付け: Gmailの非案件ラベル（`box@asai-lo.com` 通知 / `recruite@asai-lo.com` / `info@asai-lo.com` / `mailmagazines` フォルダ）、TASK-Aの「研修・活動」クラスタ（メール1,131件でも非案件）。これらは**既に非案件として観測済み** → 非案件辞書の初期値に使える。

## 5. グレーゾーンの典型と裁き方（実データ準拠）

「案件か非案件か」が**割り切れない**のがグレーゾーン。ここを類型化して**行き先を決めておく**と、現場が迷わない。

| 類型 | 例 | 行き先・裁き |
|---|---|---|
| **研修・対外活動** | 北海道包括外部監査人（1,131通）、各種セミナー | 案件ではない → **「活動」バケット**（記録は残すが受任案件に混ぜない） |
| **顧問先の定例連絡** | 顧問先からの一般相談・情報提供 | 受任案件ではない → **顧問オブジェクト**（AdvisoryContract/Consultation）へ |
| **見込み・新規相談** | まだ案件が存在しない初回問い合わせ | **新規起票候補**（§02連携）。**漏れの温床**なので特に丁寧に。捨てない |
| **自事務所が当事者** | `オークウエルネス(債権者浅井悠太)` ＝ 浅井弁護士本人が債権者 | 案件だが内部メールと混同しやすい → 当事者=自所を明示フラグ |
| **弁護士と同姓の依頼者**（誤判定注意） | `浅井眞澄_相続事件` ＝ **正規の受任案件**（姓が浅井弁護士と一致するだけ） | **非案件・内部と誤判定しない**。姓の一致で「自所案件」扱いするのは誤り。当事者は別人＝普通の依頼者として案件紐付け |
| **採用・営業・スパム** | `recruite@`宛、売り込み | **非案件確定**。探索しない |
| **判定不能** | 当事者も内容も手掛かり薄 | **未分類トレイ**（人レビュー、SLA付き） |

**裁きの肝**: 「非案件」にも**行き先がある**。ただ捨てるのではなく分類先を持つことで、後から「やっぱり案件だった」も追える＝漏れの回収経路になる。

**`non_matter_type` の粗語彙（先に決めておく / GPT監査§中5）**: 未分類に集めすぎると人間レビューが詰まるので、非案件にも最初から粗い分類語彙を持たせる。

```
non_matter_type = sales（営業）/ admin（所内事務）/ conflict_check（利益相反確認）/
                  vendor（ベンダー請求）/ hiring（採用）/ bar_association（弁護士会等）/
                  system_notice（システム通知）/ activity（研修・対外活動）/
                  advisory（顧問）/ unknown（不明）
```

## 6. 「延々探す」と「漏れる」を止める具体策

| 課題 | 具体策 |
|---|---|
| 延々探す | ゲート1で**非案件を先に確定**／ゲート2に**打ち切り条件**（候補上位スコア < 閾値、候補数0、探索N手上限）→ 未分類へ |
| 漏れる | **未分類トレイは必ず人が見る**（事務局レビュー、当日中などSLA）／案件らしさの閾値は**低め（recall寄り）**／非案件バケットも保持して事後回収可能に |
| グレーが減らない | **フィードバックで辞書を育てる**（§7）。人の確定で非案件辞書・別名辞書が太り、グレーゾーンが縮小 |
| 過信による誤り | **低精度の自動判定は自動確定しない**（事件分野auto 65.9%→write禁止 の教訓をトリアージにも適用）。グレーは人へ |

---

## 7. フィードバック・ループ（辞書が育つほど切り分けが楽になる）

- 人が **「非案件」と確定** → 非案件辞書（送信元・定型語・配信元）に追記 → 次回から早期に外せる。
- 人が **「この案件だった」と確定** → 主体辞書・別名辞書・担当priorに反映 → 案件性も紐付けも精度↑。
- **未分類から後で案件化した件** = 「漏れかけ」の貴重なサンプル → なぜ取りこぼしたかを分析し、案件性スコアを補正。

→ 初期はグレーが多く人手が要るが、**辞書が育つほど自動で切り分く割合が増える**設計にしておく。

## 8. 監視指標（この論点が改善しているかを数字で言う）

| 指標 | 意味 | 見たいこと |
|---|---|---|
| **紐付け漏れ率** | 後から案件と判明した未分類の割合 | 下がっているか（漏れ＝最重要） |
| **過剰探索率** | 非案件に費やした探索コスト／全探索 | 下がっているか（延々探しの抑制） |
| **未分類滞留時間** | 未分類トレイの平均待ち時間 | SLA内に収まっているか |
| **非案件早期確定率** | ゲート1で非案件と即確定できた割合 | 上がっているか（辞書の成熟） |

## 9. まとめ（久保さんへの引き継ぎ骨子）

1. **案件/非案件の切り分けは紐付けの前段ゲート**。ここを誤ると「延々探す」か「漏れる」の二方向に壊れる。
2. 法律事務所では **漏れ＞過剰**。だから **迷ったら未分類トレイへ（捨てない）**、ただし **探索には打ち切り条件**を持つ。
3. **2ゲート構造**（案件性ゲート→紐付けゲート）と、**非案件の積極判定（非案件辞書）** が、過剰探索を止める要。
4. **非案件にも行き先（活動/顧問/採用/通知）を持たせる**ことで、事後回収＝漏れの保険になる。
5. **フィードバックで辞書を育て**、紐付け漏れ率・過剰探索率・未分類滞留を**数値で監視**する。
