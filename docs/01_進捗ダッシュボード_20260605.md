- 作成日: 2026-06-05
- 対象: Box 全フォルダスキャン / Salesforce(LEALA) 全案件スキャン / Gmail 紐付け
- 用途: 久保さんへの進捗説明資料（客観データ編）

## 0. 全体構成（3システムの役割）

紐付けは「3つのシステムを1本の背骨（spine）で貫く」設計です。背骨は **Salesforce のレコードID（`sf_record_id`）** です。

| システム | 役割 | 本案件での位置づけ |
|---|---|---|
| **Salesforce(LEALA)** | 管制塔（案件マスタ） | 全データの突き合わせ基準。`sf_record_id` が背骨 |
| **Box** | 正本（ファイル原本・564GB） | 案件フォルダ＝1案件1フォルダ。フォルダ名で人手分類済 |
| **Gmail** | 一次データ源（メール） | ラベルで案件に紐付け。`ALO/<SF ID>_<案件名>` 形式 |

この3つを `sf_record_id` で結ぶことで、「あるメール／フォルダ／郵便物が、Salesforce上のどの案件のものか」を一意に決められる状態を目指しています。

---

## 1. Gmail 紐付け（最も定量化が進んでいる領域）

Gmail には案件ラベルが体系的に付与されており、**ラベル名そのものに Salesforce ID が埋め込まれています**。

```
ALO/a0A5h0000027BmuEAE_松井康信＿医療承継契約の審査(セカンドオピニオン)
    └─ Salesforce ID ──┘ └─ 依頼者＿事件名 ─────────────────────┘
```

### 1-1. ラベル総数 [実測]

| 区分 | 件数 |
|---|---:|
| ラベル総数 | **1,193** |
| うち案件ラベル（`ALO/` 接頭辞） | **1,181** |
| 運用・通知系ラベル（Salesforce連携通知, Box通知 等） | 12 |

### 1-2. Salesforce オブジェクト種別ごとの紐付け済み案件数 [実測]

案件ラベルに埋め込まれた SF ID の接頭辞からオブジェクト種別を判定し集計しました。

| SF IDの接頭辞 | Salesforce オブジェクト | 内容 | ラベル数 |
|---|---|---|---:|
| `a0A` | `leala__Matter__c`（受任案件） | 受任後の案件 | **780** |
| `a0D` | `leala__Consultation__c`（相談） | 受任前の相談 | **230** |
| `a1s` | 研修・活動オブジェクト | 研修・対外活動・所内業務 | **108** |
| `a03` / `a035` | 関連交渉（`leala`カスタム） | 受任案件内のサブ交渉・個別事件 | 26 |
| `a04` | 顧問契約（`leala`カスタム） | 顧問先との継続契約 | 25 |
| `001` | `Account`（取引先） | 法人・個人の取引先単位 | 9 |
| | | **合計** | **1,181** |

**読み取れること**
- 受任案件（780）＋相談（230）で全体の **約85%**。本業の案件メールはほぼ Salesforce レコードに結びついている。
- 一方で「研修・活動（108）」が一定数あり、これは後述のとおり**「案件ではないが大量のメールが溜まる」典型**で、紐付け判断の難所になっている。

## 2. Salesforce(LEALA) 全案件スキャン

Salesforce は案件マスタ（管制塔）です。オブジェクト体系は所内ナレッジで整理済みです。

### 2-1. 案件系オブジェクトの棚卸し [ナレッジ]

| オブジェクト | 表示名 | 役割 |
|---|---|---|
| `leala__Consultation__c` | 相談 | 受任前案件。クイック相談で作成 |
| `leala__Matter__c` | 受任案件 | 受任後案件。相談から変換。**`sf_record_id`＝背骨** |
| `leala__AdvisoryAcquisition__c` | 顧問獲得 | 顧問契約前 |
| `leala__AdvisoryContract__c` | 顧問契約 | 顧問契約後 |
| `leala__AdvisoryConsultation__c` | 顧問相談 | 顧問先からの相談 |
| `Account` / `Contact` | 取引先 / 取引先責任者 | 全関係者の入れ物 |
| `leala__RelatedContact__c` | 関係者 | 案件の関係者 |
| `leala__OpponentParty__c` | 相手方 | 相手方（反復相手方分析用） |
| `leala__JurisdictionCourt__c` | 管轄裁判所 | 事件番号 `court_case_key` |
| `leala__PostalMatter__c` | 郵便物 | **最もクリーンな行動データ源** |
| `leala__Procedure__c` | 手続 | 1案件にN手続（交渉→調停→訴訟） |

### 2-2. 相談 → 受任案件 の昇格紐付け（TASK-B「Junior反映」）[実測・抽出]

「相談（`a0D`）」が「受任案件（`a0A`）」に昇格したペアの突き合わせワークシート（3分割）。確認済みの行は **反映済「〇」** が立っています。

| 観点 | 内容（代表例） |
|---|---|
| 基本ペア | `相談ID(a0D…)` ↔ `受任ID(a0A…)` を1対1で対応付け |
| 名称が変わる例 | 杉本麻千子: 相談「法律相談」→ 受任「遺言無効確認等請求事件」 |
| 別案件化の例 | HIRAKUホールディングス: 相談「事業再生(私的整理)」→ 受任「破産申立事件」（備考: 別案件が受任案件に） |
| 受任なしの例 | アダチ: 相談のみ（備考: 受任案件紐づけなし） |
| 地理での分岐 | ソーケン: 相談「M&A@埼玉」と 受任「M&A@千葉」が別レコード |

→ 大半は `〇`（反映済）。残課題は「相談と受任で名称・事件性が変わるケース」の確認。

### 2-3. 案件名特定ワークシート（TASK-A「案件名特定」）[実測・抽出]

メールが大量に溜まっているのに案件名が即断できない SF レコードを抽出し、4分割で人手確認したリスト。区分は主に **研修・活動** と **取引先** です。

| 状態 | 意味 | 代表例 |
|---|---|---|
| `active` | 進行中 | 商工会議所 新入社員研修 / 研修＿近畿地区巡回研修会 |
| `dormant` | 休眠（直近活動あり） | 北海道包括外部監査人（メール1,131件） / AIシステム導入検討 |
| `cold` | 沈静化 | 事業再生連載企画 / 事務員雇用案件 |
| `archived` | 終了・保管 | 破産ストーリー本執筆 / 民事委員会 |

**読み取れること**: メール件数が多い順の上位に「**北海道包括外部監査人（1,131件）**」「事業再生連載企画（555件）」など**研修・対外活動**が並ぶ。これらは案件ではないため、件数だけで案件性を判断できないことを示す実例。

## 3. Box 全フォルダスキャン

Box は正本（564GB）。案件フォルダの分類構造が **Salesforce のオブジェクト体系と一致** しているのが特徴です。

### 3-1. 案件フォルダ階層 [実測]

```
★LEALA/
├ 01浅井LEALA/      ┐
├ 02武市LEALA/      ├ 弁護士3名でフォルダを分割
└ 03宮澤LEALA/      ┘  （2026-04 宮澤先生 join）
   ├ 1受任案件     ← leala__Matter__c に対応
   ├ 2相談         ← leala__Consultation__c に対応
   ├ 3裁判外業務
   ├ 4顧問契約     ← leala__AdvisoryContract__c
   ├ 5顧問相談     ← leala__AdvisoryConsultation__c
   ├ 6受講研修資料 ← 研修・活動(a1s) に対応
   └ 7烏丸事務所
```

→ **Box のフォルダ分類＝Salesforce のオブジェクト分類** という対応が取れている。これが紐付けの土台。

### 3-2. 浅井弁護士フォルダの案件数 [実測]

| カテゴリ | フォルダ数 | 備考 |
|---|---:|---|
| 1受任案件 | 70エントリ（うち案件フォルダ 約68） | ほか「既済」「出力確認テスト」を含む |
| 2相談 | 53エントリ（うち案件フォルダ 約49） | ほか「別案件移行済」「既済」等を含む |

> 武市・宮澤フォルダも同一構造。所全体の案件フォルダ総数はこの3名分の合計（次回スキャンで全弁護士分を集計予定）。

### 3-3. フォルダ命名規則 [実測]

```
く_久保浩二_損害賠償請求事件(交通事故)
│ └─ 依頼者名 ─┘ └─ 事件名 ──────────┘
└ 五十音プレフィックス（あ/い/う…）
```

Gmail ラベルの `依頼者＿事件名` と同じ構造。**両者を依頼者名・事件名で機械照合できる**のが紐付けの肝。
