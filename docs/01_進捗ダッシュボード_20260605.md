## 3. 紐付けの考え方＝9パターン(A〜J)〔本体ルール v1.1〕

相談・受任・関連交渉・顧問・取引先の関係を、9類型に整理済み（詳細は `02`）。

| Pattern | 名称 | 例 |
|---|---|---|
| A | 1:1 クリーンハンドオフ（相談1→受任1） | アダチ・ハナフル・京ノ・東洋化学（約6割） |
| B | 1:N 分岐（法人複数＋代表者個人） | HIRAKU・シマプリ・田中外士夫商店・公建/奥村家 |
| C | 受任先発・相談後付け（独立扱い） | 福永孝文＿法律相談 |
| D | 切り替え漏れ・相談側偏在 | シマプリ（相談184 vs 受任11） |
| E | SF未起票/活動なし（**規模大：全体の約30%**） | 河野町子GL・島内正弘個人破産 |
| F | 顧問契約は派生親でなく「土壌」 | 龍村美術織物・かね正・きゅうべえ |
| G | 関連交渉(`a03*`)＝受任のサブ事件 | 隈元建設・目片鉄工所 |
| H | 取引先(`001*`)＝マスタ・案件と1:N | シマプリAccount |
| I | 相談で終結（受任に至らず） | 高雄倶楽部・五和工業（31件） |
| J | 関連交渉→受任への昇格 | 龍村美術織物・売掛金回収事件 |

**重要**: 相談⇄受任は **多対多(N:M)**。1:1は理想だが少数派で、法的整理では「法人＋代表者個人」の枝分かれが常態。

## 5. 全数検証で確かめた「当事務所ならでは」の実例知見

机上論ではなく、**実データ・実メールで確認**した、ALOの業務特性に根ざす紐付けの勘所。

| 知見 | 実例 | 仕組みへの含意 |
|---|---|---|
| **離婚は旧姓で立件されることがある** | Box=木戸美紗（婚氏）／SF=浅山美紗（旧姓）。離婚で旧姓に復するため | 別姓は正規化では繋がらない。**分野(離婚/相続/改名)で旧姓ペアを想定**＋名寄せ台帳 |
| **同一依頼者に複数案件（B/N:M）** | 田中外士夫商店3件・龍村2件・HIRAKUグループ多数 | 依頼者一致だけで確定しない。事件名/相手方で二段確認 |
| **関連交渉は受任のサブ（G）** | 隈元建設の松尾対応・広和建設請負代金 | `a03*`は独立受任でなく親受任にリンク |
| **協業先は非案件** | 荒木先生＝公認会計士（M&A協業FA） | 当事者でない協業者は案件化しない。同姓異人に注意 |
| **利益相反確認＝消去法の担当シグナル** | @浅井・@宮澤に確認→消去法で武市担当（Box所在と一致） | メンションの種類で重み付け。確認宛先＝非担当 |
| **表記ゆれ（中黒・等）で突合が外れる** | エムイー／エム・イー開発技研 | 正規化に中黒・区切り・五十音・敬称の吸収が必須 |
| **Box案件フォルダに背骨IDが無い** | 木戸/浅山が名前でしか繋がらなかった根因 | **Boxフォルダにも`sf_record_id`を付与**すれば名称食い違いに強くなる（最優先級） |

## 6. 久保さんへの要約（1枚）

1. **やり方が堅い**: 取り違え事故を機に、Salesforce 319件を**全数検証**。確定40件＋9パターン化＋全スキャン＋DDLまで到達。
2. **いまの位置**: 設計・ルール・確定データ・Box全スキャンは完了。**P1（SF項目棚卸し）進行中**。残務は事務局のHITLタスク（TASK-A/D/E等）。
3. **仕組みの肝**: 相談⇄受任は**N:M**。紐付けは「正解を背骨ID(`sf_record_id`)で固定」し、メール/郵便/電話等は周辺シグナルで案件に寄せる。
4. **ALOならではの難所**は実データで実証済み（旧姓立件・複数案件・関連交渉・協業先・利益相反の消去法・Box背骨ID欠落）。これが「業務特性に沿った紐付け」の中身。

> 詳細の判断ロジックは `02`、精度を上げる設計材料は `03`〜`05`、突合の客観データは `06`／`data/` を参照。**確定データ・本体は `案件紐付け作業_20260520/` が正本。**
