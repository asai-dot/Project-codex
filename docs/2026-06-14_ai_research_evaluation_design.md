# AIリーガルリサーチ 評価設計 ──「人より良い」を測れる形にする（v0.1）

- 作成: 2026-06-14 / Claude（リモートセッション・浅井さん指示）
- 親文書: `docs/2026-06-14_ai_ready_legal_research_methodology.md`（§7「『人より良い』をどう測るか」を本書で展開）
- 位置づけ: ALR(AIリーガルリサーチ) が HLR(人手リサーチ) を上回ったと**言えるための測り方**の設計。実装でなく評価方法の設計。
- governance: 設計のみ。モデル実行・RAG・embedding・外部共有は HOLD。本書はベンチ/ルーブリック/正解構築手順の定義に留める。

## 0. なぜ評価を先に設計するか

「人より良いリサーチがありそう」は仮説。**測れない改善は改善でない**。先に物差しを決めると、(1) どの工程に投資すべきかが分かり、(2) データ構造への要求が逆算でき（各指標は特定のデータを要求する）、(3) お目付け役・依頼者・自分自身への説明責任が果たせる。評価設計は同時に**データ設計の仕様書**になる。

「より良い」は単一スカラーではない。**8つの構成概念へ分解**して各々を測る。

## 1.「より良い」の分解 ── 8構成概念と指標

| # | 構成概念 | 問い | 指標（operationalization） | これが効く理由 |
|---|---|---|---|---|
| C1 | **網羅 / Recall** | 関連する典拠を取りこぼさないか | プール法による recall@k（gold＋AI＋人の和集合に対する関連被覆率） | 閉じたコーパスゆえ「全部見た」が定義可能＝人の最大の弱点 |
| C2 | **精度 / Precision・ノイズ税** | 出したもののうち的中はどれだけか | precision、無関係ヒットに人が費やした時間（noise tax） | 量だけ出すのは逆に人を疲れさせる |
| C3 | **反対線の網羅 / Contrary-authority recall** | 反対説・不利な裁判例・最新改正による失効を当てに行ったか | gold の contrary set に対する被覆率 | **ALRの最大の差別化点**。人が一番サボる工程 |
| C4 | **接地忠実度 / Grounding fidelity** | 全命題に正しい検証可能な出典があるか | citation-supported率／citation-correct率（引用箇所が実際にそう言っているか）／**幻覚率** | ALR最大のリスクを定量化 |
| C5 | **時間正しさ / Currency** | 改正・廃止・判例変更を踏まえているか | stale-law エラー率（失効/旧法を現行として扱った率） | 法は時点依存。誤れば致命 |
| C6 | **問いを超えた論点出し / Issue-spotting beyond prompt** | 聞かれていない関連論点を出せたか | novel-issue yield（新規論点数）×その validated-relevance率 | **ボトルネック破壊の本体**＝想起依存の解除 |
| C7 | **判断readiness / 人の手間** | 弁護士が「自分の名前で出せる」まで何分か | 人レビュー時間、fileable までの編集距離、採用率 | 律速が「判断」へ移ったことを測る |
| C8 | **較正 / 正直さ** | 「コーパス外/不明」を正しく言えるか | known-unknown 正答率、overconfidence率 | 知らないことを知らないと言える設計か |

> 設計判断: **C4(接地)・C5(時間)・C8(較正) は“門前”指標**。これらに違反した出力は、どれだけ流暢でも**失格扱い**にしてから C1–C3・C6–C7 を採点する。幻覚した名文より、正直で出典の正しい素朴な答えを上に置く。

## 2. 正解(ground truth)の問題と4つの戦略

法律は答えが割れる領域。完全な正解は作れない。タスク種別で**正解の作り方を変える**。

1. **アンカー型タスク（客観的正解が引ける）**: 「日付Dにおける支配条文と現行条文」「コーパス内で論点Xを扱う最高裁判例の列挙」「代表的体系書の該当節」。閉じたコーパス内なら答えが**検証可能**。正解はコーパスから機械＋人で確定。
2. **プール法による recall（完全関連判定が無理なとき）**: AI＋複数の人＋既知 gold が出した典拠を**プール**し、専門家が各々に関連/非関連を付与。recall はプールに対して計算（IRの標準手法）。
3. **プロセス正解（結論が割れるとき）**: 結論でなく**「論点マップ＋典拠集合＋反対線」**を gold とする。上席2名以上の合議＋裁定で構築。評価は「人が踏むべき道を踏んだか」。
4. **参照なし＋専門家審判（統合品質）**: AIブリーフ vs 人(アソシエイト)ブリーフを**ブラインド対比**。上席パートナーがルーブリックで採点（どちらがAIか伏せる）。

## 3. ベンチマーク設計

### 3.1 タスク分類（最低この5系統を層化）
- **T1 条文ルックアップ/時間軸**: 現行条文・改正年表・施行時点（→ C5 主眼）。
- **T2 ドクトリン・サーベイ**: 論点Xの学説・体系書節・主要判例の俯瞰（→ C1/C6）。
- **T3 反対説/不利authority探索**: ある立場に対する contrary line（→ C3）。
- **T4 判例系譜の追跡**: ある論点の裁判例の流れ・射程（→ C1/C5）。
- **T5 起案支援**: 主張構成に必要な典拠の供給（→ C2/C7）。

### 3.2 各タスクの gold pack（正解パック）
1タスクにつき: 問い／日付コンテキスト／**関連典拠集合**（条文・判例ID・文献TOC locator）／**反対線集合**／**論点マップ**／難易度ラベル／構築者と合議記録。

### 3.3 規模（PoC→本番）
- PoC: 各系統2問＝**10問**（手構築・本書の§8）。
- v1: 各系統10問＝50問、gold は上席2名合議。
- 母集団: まず**3サイト共通501冊＋それらが引く判例**（前回再分析の overlap）に限定し、コーパス境界を統制。

## 4. 比較プロトコル ── 2つのフレーミングを併記

**同条件比較と現実比較は別物**。両方を測り、混同しない。

- **F-A（公平条件）**: 人とAIに**同じ閉じたコーパス**を与える。能力の純差を測る。research の上限を見る。
- **F-B（現実条件）**: 人は**普段のツール**（有償DB・記憶・同僚）、AIは閉じたコーパス。**運用上意味があるのはこちら**。「導入して効くか」を測る。

手順（1タスク）:
1. gold pack を伏せる。
2. AIが出典付きブリーフを生成（門前チェック→採点）。
3. 人弁護士が現実的時間予算でブリーフ作成。
4. 出力をブラインド化し、上席が C1–C8 をルーブリック採点＋プール関連判定。
5. タスク毎に集計、信頼区間つきで報告。
- 統制: 時間予算の明示、問題順のカウンターバランス、審判の AI/人 盲検、利益相反（自分のAIを自分で甘く見ない＝お目付け役レビューに載せる）。

## 5. アンビエント・モードの評価（ボトルネック仮説の本丸）

固定の問いへの回答だけ測っても、製品の主張（**人が気づかない論点を拾う**）は測れない。会議トランスクリプトを入力に:
- **issue-surfacing recall**: その会議で*出すべきだった*論点（事後に上席が確定）のうち、システムが先に挙げた割合。
- **human-noticed baseline**: 同会議でその場の弁護士が気づいた論点。これを基準線に、**Δ = システムが拾い人が落とした論点**を主指標に。
- **lead time**: 論点が必要になる前にどれだけ早く出せたか（夜間バッチでも可ゆえ速度でなく**先回り**を測る）。
- **noise/疲労**: 誤検知率と、人が捨てるのに要した時間（C2 と連動。閾値設計の評価）。

→ ここで「速さは要らない／網羅と先回りが価値」という浅井さんの直観が、Δと lead time として数値になる。

## 6. 門前チェック（trust contract 準拠）とコスト会計

- **門前（不合格なら採点せず差し戻し）**: grounded-only（無典拠命題ゼロ）／ロケータ必須（書名+頁・判例ID・条番号）／コーパス外申告あり。
- **コスト会計**: トークン/¥、エージェント数、所要時間。**C1–C7 の改善を“1ブリーフあたりコスト”で割った費用対効果**を併記（マルチエージェントは高い。どこに効かせるかの判断材料）。

---

## 7. データ構造への含意（各指標 → 要求データ）

評価設計はそのままデータ要求になる。「測りたいなら、これを持て」:

| 指標 | 必要なデータ/構造 |
|---|---|
| C1 網羅 | TOC locator の検索可能化・用語グラフ（問い拡張） |
| C3 反対線 | **引用グラフ**（書籍↔判例↔判例）・判例の treatment（参照/批判/変更） |
| C4 接地 | TOC locator＋**文献本文(OCR)**・判例本文・条文の安定参照 |
| C5 時間 | 法令 currency（施行/改正時点）・判例の overrule 関係 |
| C6 論点出し | 「論点(issue/doctrine)」中間層・用語↔条文↔判例↔文献節リンク |
| C7 手間 | 出典の格付け（権威重み）・横断合議スコア（信頼度） |
| C8 較正 | コーパス境界メタ（収録/未収録・OCR未取得・規約制限の明示） |

→ 評価を真面目にやろうとすると、前回の＋α（本文OCR・引用グラフ・用語グラフ・論点層・重み付け）が**全部「測るために要る」と再導出される**。評価とデータ設計は同じ問題の表裏。

## 8. HOLD と整合する PoC（設計＋手構築のみ）

実行系は HOLD なので、**人手で1問“通し”を作ってルーブリックを検証**する:
- **P1**: 3サイト共通501冊から T2/T3 の各1問を選び、gold pack（関連典拠＋反対線＋論点マップ）を手構築。
- **P2**: その1問に対する「理想ブリーフ」を手で書き、C1–C8 ルーブリックで自己採点 → ルーブリックの穴を出す。
- **P3**: 同問題を普段のやり方で解いた所要時間・典拠数を記録（F-B 基準線の最初の1点）。
- **P4**: 上記を `eval/` 配下のスキーマ（タスク・gold・ルーブリック・採点表）として固定。モデル実行は別承認後。

## 9. 残論点
- gold 構築の**コスト**（上席2名合議は高い）。どこまで自動下書き＋人裁定にするか。
- 「結論が割れる問い」で**プロセス正解**は本当に公平か（道は複数ありうる）。
- アンビエント評価の **issue 母集合**を事後に誰がどう確定するか（後知恵バイアス）。
- F-A と F-B の**どちらを意思決定に使うか**（上限 vs 現実）。
- 評価自体の**自己採点バイアス**（自分のAIを甘く見ない統制＝お目付け役の独立監査に載せる）。
