# AIレディーなリーガルリサーチ ── 手法の変化と設計思想（v0.1 思考メモ）

- 作成: 2026-06-14 / Claude（リモートセッション・浅井さん指示）
- 位置づけ: 「人がやるより良いリサーチ」をクローズドな法律ドメインで実現するための**思考の記録**。実装ではなく方針。
- 前提資産（既に手元にあるもの）: 法令（e-Gov 法令・定義）／判例（D1-Law 141,378件）／法律図書のメタ＋詳細TOC（LION BOLT・弁コム・legal-library 計約31,000冊、TOCノード約164万）／法律雑誌（LIC 4誌の解説本文7,394件・legallib雑誌422号）／法律用語・語彙（term_dict・legal_term_dictionary.json）／**ミーティング→案件パイプライン**（meeting_to_case・meeting_todo_writer・meeting_analysis）
- 未取得: **文献の本文**（OCRの壁）。詳細TOCまでは到達＝「生データにかなり近い」状態。
- governance: 本書は設計検討のみ。RAG/embedding/外部共有は HOLD 中。本書は実データを動かさない。

## 0. テーゼ（3行）

1. **質的転換の本体は「速さ」ではなく「想起と網羅の限界の解除」にある。** 人間のリサーチは作業記憶（同時に数件）・逐次読解・**想起依存の論点出し**（思いついた論点しか調べられない）に縛られる。AIはクローズド・コーパス上で網羅・並列・潜在シグナルからの論点生成ができる。
2. **ボトルネックは「1対1チャット」という同期・受動の構造そのもの。** 人が(1)調べる必要に気づき(2)問いを立てる、この前段が律速。AIレディー化とは、**リサーチを“チャットの1ターン”から“常時走る背景プロセス”へ**反転させること。
3. **クローズド・ドメインこそAIが人を超えうる場所。** 出典が確定し再配布規約も明確な閉じたコーパスでは、全命題に出典ロケータを刻み、敵対的に反証し、コーパス境界を正直に申告する設計が可能＝**人より検証可能で信頼できるリサーチ**にできる。

## 1. ヒューマン・リーガルリサーチ(HLR) と AIリーガルリサーチ(ALR) の質的差

「速いか遅いか」ではなく、**何が制約か**が違う。軸ごとに整理する。

| 軸 | HLR の制約 | ALR が変える点 | 必要な手当て |
|---|---|---|---|
| **網羅(recall)** | サンプリング（主要書数冊＋勘）。読んでない本は論点に上がらない | 約164万TOCノードを節レベルで全走査。読まずに「どの本の何章が該当か」 | TOCロケータの検索可能化・語彙による問い拡張 |
| **並列(parallelism)** | 1人が逐次。サブ論点は順番待ち | 論点分解→多エージェント並走 | プランナ＋並列リトリーバ＋統合 |
| **論点出し(issue spotting)** | **想起依存**。気づいた論点しか調べない | 潜在シグナルから論点を生成（「会議でXに触れた→閉じたコーパス上でXは反対説Yと共起」） | 用語抽出→候補論点オブジェクト生成 |
| **反証(falsification)** | 自分の仮説を自分で十分には潰せない | 敵対的エージェントが反対線（contrary authority）を専門に探す | devil's-advocate 役の常設 |
| **追跡可能性(traceability)** | 記憶からの引用は検証しづらい | 全命題に出典ロケータ（書名+頁／判例ID／条番号） | grounding 契約（出典なき主張を出さない） |
| **判断(judgment)** | ここが人間の価値 | AIは*代替しない*。素材と論点を上げ、人が決める | human-in-the-loop の設計 |

→ 質的転換の要約: **「人が問い、AIが取りに行く（＝速い司書）」から「AIが論点候補と証拠を連続生成し、人が判断する（＝疲れないアソシエイト＋人はパートナー）」へ。** 律速が「検索のスループット」から「判断のスループット」へ移る。人間は価値連鎖の上流（フレーミングと意思決定）へ押し上げられる。

### 1.1 「人より良い」が成立する条件
速度は要らない（人の反射は遅くてよい）。「人より良い」が出るのは次の3つが揃ったとき:
- **網羅** ── 人が見落とす本・章・判例まで届く（閉じたコーパスなので“全部見た”が言える）。
- **反証** ── 自説に都合の悪い線を必ず当てに行く（人が一番サボる工程）。
- **接地** ── 全部に出典がつき、コーパスの外は「外」と正直に言う（幻覚の封じ込め）。
この3点は、実は**人間が構造的に苦手な工程**。だからこそAIが上回りうる。

## 2. ボトルネックの再定義 ── 受動チャットから常時リサーチへ

浅井さんの中核的直観の定式化:

- **現状(1対1チャット)** は同期・受動。人が「①必要に気づく → ②問いを立てる → ③待つ → ④読む」。**律速は①②**で、これは想起依存＝人間の弱点そのもの。チャットUIはこの弱点を温存している。
- **AIレディー構造** はこれを反転する:

```
[常時の文脈入力]            [連続処理]                    [非同期の成果]
会議の発話・議事録の用語  → 用語/争点抽出 → 候補論点生成 → リサーチ・ジョブ → 出典付きブリーフ
意図的に打ち込んだメモ        （閉じたコーパスを横断）                      （人が後でレビュー・判断）
案件ファイル
```

- ここで**速度は制約ではない**（夜間バッチでも翌朝で十分なケースが多い）。制約は**網羅と関連性**。リサーチを「チャットの1ターン」ではなく「案件に常駐する背景プロセス」として設計する。
- **基盤は既にある**: meeting_to_case / meeting_todo_writer / meeting_analysis（会議→案件→ToDo）と term_dict。ここに「候補論点の生成と、それを満たすリサーチ・スキルの起動」を接ぐのが次の一歩。トリガ層の素材は揃っている。

## 3. AIレディー・リサーチの参照アーキテクチャ（6層）

下から積む。★=既にある / ◇=設計HOLD中 / ＋=新規に要設計。

1. **コーパス層** ★ ── 法令・判例・書籍メタ＋TOC・雑誌本文・用語。
   - 不足: **文献本文（OCR）**、および前回再分析で挙げた射影/重み（ジャンル・OCR品質・あらすじ・引用リンク）。
2. **索引・検索層** ◇＋ ── ハイブリッド検索。
   - 語彙的（用語・TOC見出し）＋意味的（embedding, HOLD）＋構造的（引用グラフ・条文相互参照）。閉じたコーパスゆえ高精度＆出典確定が“堀”。
3. **知識・オーソリティ層** ◇ ── 正準著作の同一性、引用エッジ、用語グラフ（broader/narrower）、法令時間軸（DD-LAWTIME）。
4. **スキル・ツール層** ＋ ── **ゴール形のリサーチ・スキルをツール化**（汎用検索ではなく、目的別の合成）:
   - 「反対説・contrary authority を探す」
   - 「ある論点の判例の流れ（裁判例の系譜）を辿る」
   - 「支配的条文＋その代表的コンメンタール節（TOC経由）＋引用判例を一括取得」
   - 「ある法令の改正年表を作る」
   - 「論点Xの主要体系書の該当節を特定し、そこが引く判例まで降りる」
   - 各スキルは「コーパス上の決まった経路を辿る関数」。AIはこれを**部品として組む**。
5. **エージェント・オーケストレーション層** ＋ ── deep-research型の型（fan-out→敵対的検証→出典付き統合）がそのまま使える:
   - プランナ（案件の問いをサブ論点へ分解）／並列リトリーバ（各サブ論点）／**検証者（反対線を専門に探す）**／統合者（出典付きブリーフ）。
6. **トリガ・アンビエント層** ＋★ ── 会議・議事録・メモ→用語抽出→候補論点→ジョブ→ブリーフ→人のレビュー。基盤(meeting pipeline)は★、論点生成と起動が＋。

## 4. 信頼の契約（ALR最大のリスク＝自信満々の誤りを封じる）

クローズドだからこそ強制できる規律。これがALRを“人より信頼できる”側に倒す:

- **接地のみ(grounded-only)**: コーパスに典拠のない命題を出力しない。
- **ロケータ必須**: 各命題に「書名+頁(TOC locator)／判例ID／条番号」を付す。
- **敵対的検証(adversarial pass)**: 結論を一度反証しに行く工程を必須化（contrary authority・反対説・最新改正による失効）。
- **境界の正直さ(known-unknown)**: コーパスの外は「外」と言う。OCR未取得・収録外・規約制限を明示。
- **出典の格付け**: source priority（manual > NDL > publisher > toc_pdf > legallib > openbd > … > OCR）を信頼度として伝播。
- **時間正しさ**: 法令の currency（施行/改正時点）を判断に必ず絡める。

## 5. データ構造へのフィードバック（＋α / −α / 重み付け）

「リサーチ手法をAIレディーにする」と決めると、収録情報と構造に次が要求される:

### ＋α（足す）
- **文献本文（OCR）** ── 最大の欠落。節（TOC locator）に本文を結びつければ「節→本文→引用判例」が一気通貫。
- **引用エッジ** ── 書籍→判例（弁コム 428,217リンク）、判例→法令、書籍→書籍。**引用グラフ**は権威の重み付けと論点伝播の骨格。
- **用語グラフ** ── 用語の broader/narrower/関連、用語↔条文↔判例↔文献節のクロスリンク。問い拡張と論点生成の燃料。
- **embedding** ── 意味検索（HOLD。DD-EMBED-001 accept後）。
- **「論点/ドクトリン」層** ── 用語・条文・判例・文献節を束ねる中間オブジェクト（リサーチの最小単位を「文献」でなく「論点」にする）。
- **候補論点オブジェクト** ── 会議用語→生成される未確定の論点。confidence と出典つき。

### −α（足さない/避ける）
- 出典を消す過正規化をしない（provenance は資産）。
- 版を束ねない（DDL-20260428-01）。
- 短命な gated URL を安定参照のように持たない。

### 重み付け（weighting）
- **出典権威**（上記 priority）／**新しさ**（法令 currency・出版年）／**OCR精度**（accuracy_rank A/B/C）／**横断合議**（複数ソースが一致＝信頼度↑。前回の「横串ジャンル」を一般化し“クロスソース一致度”を confidence に）。

## 6. 近接アクション（HOLD と整合する“設計のみ”）

実DB書込・embedding・外部共有なしで着手できるもの:

- **S1. リサーチ・スキル・カタログを書く** ── §3-4 の各スキルを「入力／コーパス経路／出力／出典契約」で定義（まず5本）。
- **S2. 「候補論点」オブジェクトと 会議→論点トリガ仕様** ── meeting pipeline の出力から用語を拾い、候補論点を起こす最小スキーマ。
- **S3. 接地＋敵対的検証プロトコル** ── grounded-only / ロケータ必須 / 反証パス / 境界申告 の契約文書。
- **S4. 重み付けモデル仕様** ── 権威×新しさ×OCR精度×合議 を1つのスコアに合成する式の草案。
- **S5. 紙上PoC** ── 3サイト共通501冊＋該当判例で、「ある論点について“反対説まで含む出典付きブリーフ”を1本」手で組んで、人のリサーチと比較する評価設計だけ先に固める（実行は別承認）。

## 7. 論点として残すもの（次に深める問い）

- リサーチの最小単位を「文献」から「論点(issue/doctrine)」へ移すと、DBの中心テーブルは何になるか。
- アンビエント生成された論点の**ノイズ管理**（人を疲れさせない閾値・束ね方）。
- 「人より良い」を**どう測るか**（網羅率・反対線の発見率・出典正答率・人レビュー時間の短縮）。
- マルチエージェントの**コスト/便益**（並列はトークン的に高い。どこに効かせるか）。
- クローズド・コーパスの**堀**としての価値（一般LLMが見られない閉じた高精度＋出典確定）と、その維持（規約・更新）。
