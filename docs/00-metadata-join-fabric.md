# 接合ファブリック ＋ 共通 resolver 背骨（Fork 0）

> 法令・判例・文献・雑誌・語彙・手続・書式が、**本文なしでメタデータ同士の接合だけで
> 精度を出す**ための全体モデル。各フォーク（1〜4）はこの背骨に別エッジを足す作業。

## 1. なぜ“本文なし”で効くのか

法律ドメイン特有の3つの理由でメタデータ接合が効く。

1. **目次・論文タイトルは「人が圧縮した要約」** — 本文より信号密度が高くノイズが少ない。
   章見出し「時効の中断と停止」1行が、本文20ページより主題判定に効く。
2. **法律ドメインは“構造化された自然言語”** — ほぼ正規キーが既に埋まっている。
   `法令名＋条項` / 判例の `事件番号・裁判年月日＋裁判所・判例集出典` / `ISBN` / `著者名`。
3. **共起だけで辺が引ける** — TOC見出しが条文・判例番号を含めば、本文を読まずに
   「この章 ↔ この条文」の辺が張れる。

## 2. 7ノードの接合ファブリック（実際の join キー）

```
        ┌─────────[語彙/term_dict]─────────┐   ← 全辺の“接着剤”（見出し→主題正規化）
        │              │            │       │
     [法令]────条文番号───[文献TOC]──著者──[雑誌/論文]
     法令ID         (ISBN↔book_id     {title, authors,
     名+条項         =resolver)        掲載誌+巻号+page}
        │              │
     条文├──引用──[判例]              [手続]───[書式]
        事件番号/年月日+裁判所         ↑紐付き先 = 法令条項 + 文献の章節
```

| ノード | 正規キー | 主な接続先 |
|---|---|---|
| 法令（e-gov） | 法令ID / 法令名+条項 | 文献TOC・判例・手続・書式 |
| 判例 | 事件番号 / 裁判年月日+裁判所 / 判例集出典 | 法令条文・文献TOC |
| 文献（books） | ISBN（↔ legallib book_id は resolver 経由） | 法令・判例・雑誌（著者）・語彙 |
| 雑誌 / 論文 | book_id ＋ 論文entity `{title, authors, 掲載誌+巻号+page}` | 文献（著者）・法令・判例 |
| 語彙（term_dict） | 用語 → 別名 / 上位下位 | **全ノード（背骨）** |
| 手続 | （実体未確認・フロンティア） | 法令条項 + 文献の章節 |
| 書式 | （実体未確認・フロンティア） | 手続 + 法令条項 |

- **語彙が背骨**：見出しの表記ゆれを主題に正規化できると全ノードが同じ座標系に乗る。
- **著者**が文献↔雑誌を縫い、**条文番号**が法令↔文献↔判例を縫う。
- **手続・書式**は現状いちばん証拠が薄い。条文＋文献章節にぶら下げる派生ノードになりそう
  （別途、資産の棚卸しが必要）。

## 3. 精度の伸びしろは“同定”にある（留保）

メタデータ接合は**候補の recall を安く稼ぐ**のが得意。一方、precision の最後の詰めは
disambiguation が要る：

- 同名著者
- 略称条文の表記ゆれ（「民訴○条」など）
- **改正で条番号が動く**（旧条番号 → 現行条文の正規化が必要）
- 判例の複数出典

## 4. 共通 resolver 背骨

book↔ISBN で実証済みの resolver パターンを、**7ノード全部の接合に効く共通背骨**として
一般化する。book接合専用にしておくのは勿体ない。

```
candidate generation（メタ共起で安く広く）
        ↓
resolver: auto_accept / human_review / defer_new  ＋ 誤マージ0ガード
        ↓
canonical へ反映（出所 = source を必ず残す）
```

book↔ISBN での実績（2026-06-06 再走）:

- `auto_accept` 1,839（うち既merge 344 → 新規 ≈1,495）
- `human_review` 305
- `defer_new` 616（legal-library 限定＝事務所所蔵外）

この3層 + 誤マージ0ガードを、法令↔文献、判例↔文献、著者同定 等にも同じ形で適用する。

## 5. ネットワーク効果（“一気に具現化”の正体）

エッジを1本足すごとに、他のエッジの精度も上がる。線形でなく**相乗**。

- 語彙 → 判例の主題曖昧性を消す → 文献↔判例が鋭くなる
- 著者グラフ → 文献の同定を助ける → 雑誌↔文献が締まる
- 条文改正履歴 → 旧条番号を現行条文へ正規化 → 古い文献も繋がる

揃うほど加速する。だから全体像が急に像を結ぶ。

## 6. フォークとの関係

本メモ（背骨）が上位概念。各フォークは背骨に別エッジを足していく。

- **Fork 1（接合の実装設計）= 背骨そのもの**。他フォークの土台。
  特に `source` 優先順位ポリシーは先に握ると手戻りが減る。
- Fork 2（検索/RAG）・Fork 4（論文/法令リンク）は接合に依存。
- Fork 3（購入レコメンド）は接合不要で単体成立。
