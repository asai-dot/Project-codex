### 3-2. 別名辞書＋正規化を当てた再判定（→ `data/gap_rejudged.csv`）

26件に別名展開・語順・括弧内別名・敬称除去を当てて再判定した結果：

| 判定 | 件数 | 意味 |
|---|---:|---|
| ✅ **解消（確度高）** | **23** | 実はGmailラベルが存在。当初の「未検出」は**突合の取りこぼし**だった |
| ❗ **真の漏れ候補** | **2** | エムイー開発技研（受任）／ 木戸美紗（相談）。どの表記でもGmailラベルが見つからず＝**メール未ラベリングの可能性** |
| ⚠️ **要確認（フォルダ性質）** | **1** | 荒木先生。「先生」付き＝**紹介者/同業の可能性**。依頼者名が別の可能性 |

**つまり当初26件の漏れ候補は、再判定で実質2〜3件まで縮小**。残り23件は「フォルダはあり、メールも紐付いていたが、ラベル名の表記ゆれで突合が外れていた」もの。

#### ここで判明した最重要のデータ品質知見（仕組みに直結）
当初の取りこぼしの原因＝**Gmailラベルの命名が一貫していない**こと。具体的には：
1. **区切り文字の混在** ― 依頼者と事件名の区切りが全角 `＿` の場合と半角 `_` の場合がある。
2. **五十音プレフィックスの混入** ― `ALO/..._か＿門泰之＿退職…` のように、Boxフォルダ用の五十音（`か＿`）がラベル名にも入っている例。
3. **会社名＋個人名の連結** ― `エコライフフジワラ` は `エコライフフジワラ＿…` と `藤原寛治(エコライフフジワラ)＿…` の両方で出現（語順・別名違い）。
4. **敬称・括弧書き・空白**の有無。

→ **仕組み側の含意**: 突合キーは「単純一致」ではなく、**(a)区切り正規化（＿/_を統一）(b)五十音プレフィックス除去 (c)会社名↔個人名の相互別名 (d)敬称・括弧・空白除去** を施した上で照合すべき。これは §03-§5項目A（正規化ルール）と §5別名辞書の**具体的な必須要件**として確定した。

### 3-3. 真の漏れ候補を実メールで確認した結末（→ 漏れはゼロ）

残った「真の漏れ候補2件＋要確認1件」を**実際のGmailで検索**して決着させた。結論：**当初26件のうち、本当のメール紐付け漏れはゼロ**だった。

| 候補 | 実メール確認の結末 | 判定 |
|---|---|---|
| **エムイー開発技研** | `株式会社エム・イー開発技研＿地位確認等請求控訴事件` 等のラベルで**紐付け済**。Box「エムイー」とSF「エム・**イー**（中黒）」の差で突合が外れていただけ | ✅ 解消（表記ゆれ） |
| **木戸美紗** | SF/Gmail側は **`浅山美紗_法律相談`** で存在。同一案件の根拠＝Chatterの利益相反確認に「木戸美紗（依頼者）／木戸順也（相手方）／離婚案件」。**浅山が旧姓**で、**離婚で旧姓に復するため旧姓で立件**したと見られる（婚氏＝木戸／旧姓＝浅山）。実務上ありふれたパターンで、データ誤りではない | ⚠️ 解消だが**要名寄せ（旧姓⇔婚氏）** |
| **荒木先生** | araki@ark-ps.com＝**公認会計士 荒木聡（株式会社荒木パートナーズ）**。M&Aの協業FA。Box「荒木先生」相談フォルダは**協業先フォルダ**で案件ではない | ✅ 非案件（協業先） |

#### ここで判明した、正規化だけでは解けない難所（最重要）
1. **中黒「・」の有無** ― 「エムイー」と「エム・イー」。→ 正規化に**中黒の吸収**を追加（確定）。
2. **離婚案件は「旧姓で立件」されることがある（予測可能な実務パターン）** ― Box＝木戸美紗（婚氏）/ SF＝浅山美紗（旧姓）。**離婚で旧姓に復するため旧姓で案件を立てる**運用があり、これは入力誤りではなく**法律実務上の正規パターン**。同様に**改名・相続（被相続人と相続人の姓）**でも姓が割れる。**正規化（表記ゆれ吸収）では絶対に一致しない**（別の姓だから）。
   → 対策は3つ：(a)**離婚・相続・改名の分野では旧姓⇔現姓ペアを想定**し、依頼者の旧姓・通称を別名として持つ。(b)**周辺シグナルでの突合**（相手方名「木戸順也」・分野「離婚」・同時期）＝§03の3系統照合。(c)**人手の名寄せ台帳**（`dict_aliases.csv` に `浅山美紗↔木戸美紗` を旧姓⇔婚氏として収録済）。**根本解決は §3-4 のBox背骨ID**（名前に依存しなくなる）。
3. **協業先（会計士・司法書士・FA・他士業）は非案件** ― 当事者ではなく協業者。`dict_noncase.csv` に追加。**同姓異人**にも注意（本件でも「荒木聡（協業FA）」と「京都市市税事務所の荒木（別案件の連絡者）」が別人として登場）。
4. **利益相反確認の@メンションは「消去法」で担当を指すシグナル** ― 利益相反確認は**担当していない弁護士に問い合わせる**運用。本件は `@浅井 @宮澤` に確認 → **消去法で担当は武市**。実際にBoxの `02武市LEALA/相談` に `木戸美紗_法律相談` があり一致（宮澤先生は2026-04参画で案件フォルダはまだ空＝この推定とも整合）。
   → つまり@メンションは**ノイズではなくシグナル**。ただし**解釈を間違えてはいけない**：利益相反確認の宛先は「担当」ではなく「**非担当（候補から除外）**」。担当priorの出所として、弁護士別フォルダ・SF Ownerに加え、**「利益相反確認の宛先＝非担当者」を消去法に使える**（§03-Eの担当priorを補強）。一方、単なるCc/言及は弱いシグナルに留める。
5. **事件の段階差** ― Box「地位確認等請求事件」↔ SF「地位確認等請求**控訴**事件」。手続進行でラベル名が変わる（§02-4 事件番号アンカーが効く場面）。

> **久保さんへの一番のメッセージ**: 「漏れ」に見えたものは**ほぼ全て名称の不一致**であって、紐付け作業の失敗ではなかった。だからこそ **正規化ルール・別名辞書・システム間の名寄せ台帳・周辺シグナル突合**への投資が、最も効く。特に **木戸美紗＝浅山美紗のような別姓の食い違いは、名前だけでは永遠に繋がらない** ― ここに人手確認と周辺情報（相手方・分野・同時期）の価値がある。

### 3-4. 本来は「一発で繋がる」べき ― 根因はBox案件フォルダに背骨IDが無いこと

木戸美紗は **`02武市LEALA/2相談案件/き_木戸美紗_法律相談`** という**フォルダが既に存在**する。フォルダ所在だけで「**担当＝武市・区分＝相談・依頼者＝木戸美紗**」が確定している。本来なら一発でSFレコードに繋がるべき情報だ。なぜ私の突合では「漏れ候補」になったのか——

| | 背骨ID(`sf_record_id`)の有無 | SF/Gmailとの接合キー |
|---|---|---|
| Gmail案件ラベル | **あり**（`ALO/<sf_record_id>_…`） | IDで一発join |
| Box案件フォルダ | **なし**（`き_木戸美紗_法律相談`） | **依頼者名しか無い** → 木戸≠浅山で切れる |

**根因**: Box案件フォルダだけが背骨IDを持たず、SFとの唯一の接合が「依頼者名」になっている。だから名称が食い違うと繋がらない。フォルダ所在という確定情報を活かせていない。

> これは**設計の本来意図とズレている**。ALO-BRAIN にも `flow|case|LEALA(SF)→sf_record_id→Box-folder→GDrive-folder` とあり、**SFのレコードIDがBoxフォルダまで通る前提**。まだフォルダ側にID実装（＝クエリ可能化）がされていないだけ。

#### 推奨：Box案件フォルダにも背骨IDを持たせる（Gmailと同じ規約をBoxにも）
- **方法**: Boxの**メタデータ・テンプレート**（`sf_record_id` / `object_type` / `担当`）を1つ作り、各案件フォルダに付与。Boxは `set_folder_metadata`・`search_files_metadata` でメタデータ検索が可能。
- **効果**: folder ↔ SF ↔ Gmail が **背骨IDで一発join**。**木戸/浅山のような別姓の食い違いがあっても繋がる**（名前照合は補助に降格）。§3の「漏れ候補」概算が**確定値**になる。
- **運用**: 新規フォルダ作成時にSF IDを付ける運用にすれば以後は自動。既存フォルダはSF↔Box名寄せを一度通して後付け（フォルダ名の依頼者＋担当＋区分で大半が機械的に確定、残りを人手）。
- これを §03 の手戻り防止チェックリストに **「Box案件フォルダに背骨IDを付与」** として追加（最優先級）。

## 4. 分析で見つかった実データの知見（仕組みに効く発見）

1. **事務所は既に手作業のトリアージ・ラベルを運用していた** ― `案件外` / `要確認` / `誤分類` のラベルを検出。これは §04 の「非案件確定／グレー→人レビュー／誤紐付け訂正」が**既に現場運用されている実証**。仕組みはこの3状態を踏襲すればよい。
2. **「紹介元」は独立した重要カテゴリ** ― 法テラス・京都弁護士会・遺言相続センター・駅前夜間相談・欠陥住宅京都ネット が相談ラベルに頻出。**非案件ではなく「相談の入口」**。紹介元は当事者・分野の推定に効く別シグナル（特に武市Tの相談で顕著）。
3. **同一依頼者・複数案件が実在** ― 田中外士夫商店(3件)・龍村美術織物(2件)・福永孝文(受任+相談)。§02-3-3「依頼者一致だけで確定しない」の生きた根拠。`dict_attorney_prior.csv` に件数付きで収録。
4. **括弧書きが別名の宝庫** ― 旧称(旧サーモネット)・グループ(HIRAKU=旧はたご・ASUMO・コーチ)・当事者区分(個人/法人)・事件別名(長野地裁・GMO・三井案件) を自動抽出。別名辞書の良い出発点（要キュレーション）。

## 5. 辞書の使い方・育て方（手戻り防止）

- **これらは"Seed（種）"**。完璧ではなく、**人手キュレーション前提**。特に `dict_aliases.csv` は紹介元・裁判所名・事件別名が混在するので、**kind列で分類しながら精査**する。
- §03-5-7 / §04-7 の**フィードバック経路**に接続: 人が確定・訂正するたびに各辞書へ追記 → カバレッジが上がり、§3の漏れ候補が縮む。
- 次の精度向上ステップ（任意）: 別名辞書を当てて§3の26件を再判定／Salesforceと突合してSF側の正準名を確定／宮澤T・既済・裁判外・顧問フォルダを取り込み対象範囲を拡大。

## 6. まとめ

- **突合マスタ＋3辞書**で、設計（02〜05）が**実データの種**になった。久保さんはこれを取り込んで初期精度を底上げできる。
- 客観スナップショットとして現用案件フォルダの約18%（26件）が一次漏れ候補に挙がったが、再判定＋**実メール確認で真の紐付け漏れはゼロ**（§3-3）。全件、表記ゆれ・システム間の名称食い違い・協業先（非案件）のいずれかだった。
- これにより、必要なのは紐付け作業のやり直しではなく **正規化ルール（中黒・区切り・五十音・敬称）＋別名辞書＋システム間の名寄せ台帳＋周辺シグナル突合** だと**実データで裏づけられた**。特に **別姓の食い違い（木戸美紗＝浅山美紗）は名前だけでは繋がらず**、相手方・分野・担当などの周辺情報（§03）と人手確認が要る。
- 現場が既に運用する `案件外/要確認/誤分類` ラベルは、§04トリアージ設計の妥当性を裏づける。
