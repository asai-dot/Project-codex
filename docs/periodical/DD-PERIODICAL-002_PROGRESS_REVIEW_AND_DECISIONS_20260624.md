## 2. ヘッド決定（owner gate不要・私の職責で確定）
**D1. L3を「基盤完成」として凍結。** v11=97.2%で production-ready 基盤と宣言。残237誌の長尾は
`accept-as-unresolvable`（ISSN無しニュースレター/廃刊/極小誌）として受容し、**これ以上の長尾グラインドは停止方針**。
Mac Cloud Codeは現バッチ完了をもってL3を畳む（非破壊なので走り切らせてよいが、新規グラインドは追加しない）。

**D2. 別冊ジュリスト(BN01263667, 判例百選58誌/11,764記事)→ `isbn_per_issue` に確定。**
理由: 各百選版は固有ISBNを持つ書籍で、事務所本棚での実体も書籍。別冊ジュリスト通巻はメタに安定して
載らず `ncid#通巻` だと憲法百選と民法百選が同一issue_idで衝突する1万件規模リスク。ISBN単位なら構造的に解消。
→ 実行担当へ指示: 58誌を seed_bessatsu_jurist から isbn_per_issue へ再分類。

**D3. 次マイルストーン = L4記事↔issue_id接合（ORCH-ARTICLE-JOIN）を最優先に昇格。**
read-only・owner GO済・入力(D1 labeled+authority v11)即用意可。基盤が97%揃った今が着手点。
**受入基準（私が版で検査）**: article_orphan=0(authority未解決由来除く)/article_collision=0/接合被覆≥95%/
別冊ジュリストがisbn_per_issueで衝突0。これを満たせば L4 を「接合完了」と認定。

**D4. L0-L2(OCR)は article-join 認定後に着手。** OCRは1誌数号パイロット→私の境界/conf監査合格ゲート→全量。
バルクOCR後処理はQEN(ローカルちゃん/Ollama)に回す。初出(pacsigny)はL4接合の上に乗せる。
