# 紐付け精度を上げる要素の洗い出し ― 仕組み構築の事前準備

- 作成日: 2026-06-05
- 用途: 久保さんが「紐付けの仕組み」を作る前に渡す**設計材料**（手戻りを減らすための事前準備）
- 立場: **作るのは久保さん／使うのはALO**。良いものを作るために、精度の上がる要素・下げる要素を先に棚卸ししておく。
- 読み方: これは確定仕様ではなく「実装前に決めておくと手戻りが減るチェックリスト」。**[実績]** はこれまで実際にやって効いた打ち手、**[計測]** は所内で測った精度値、**[設計提案]** は仕組みに先に織り込むべき土台。

---

## 1. これまでの試行錯誤で「効いた」ポイント（実績ベース）

> ここが本資料の核。**実際にやって精度が上がった打ち手**を、なぜ効いたかと一緒に残す。久保さんが同じ試行錯誤を繰り返さないために。

| # | やったこと | なぜ精度が上がったか | 効き所 |
|---|---|---|---|
| 1 | **Salesforce IDをラベル/フォルダ名に埋め込んだ** `ALO/<SF ID>_依頼者＿事件名` [実績] | 一度貼れば名寄せが不要になる。表記ゆれ・同名問題を**根本から回避**できる「正解の固定」 | **最大の精度レバー**。確定済みデータを増やすほど全体精度が上がる |
| 2 | **`依頼者＿事件名` の区切り規則を統一** [実績] | 「＿」で機械分割し、依頼者キーと事件キーを**別々に照合**できるようにした。依頼者一致と事件一致を分けて重み付けできる | 半自動マッチの土台 |
| 3 | **自動分類の精度を測り、低いものは自動書き込みを禁止した** ― 事件分野の自動付与は **65.9%** で精度不足 → **write禁止**にした [計測] | 「精度の低い自動化はノイズを撒く」。サジェスト止まりにして人が確定する運用に切替えた | **誤紐付けの量産を防いだ**。精度しきい値の考え方の原型 |
| 4 | **浮遊データを隔離した** ― 担当所員リレーションの **96.8%が浮遊**（案件に紐付かない）→ 隔離対象に [計測] | ノイズ源をマスタから外すと、残ったデータの信頼度が上がる | 照合母集合のクリーン化 |
| 5 | **研修・対外活動の大量メールを「案件」から切り離した**（TASK-A 案件名特定）[実績] | メール1,131件のように**件数が多くても案件ではない**クラスタを区分。「件数の罠」を回避 | 非案件の混入防止 |
| 6 | **相談⇄受任のペア表を人手で作り、名称ズレを吸収した**（TASK-B Junior反映）[実績] | 「法律相談」→「○○請求事件」のように名称が変わっても**同一依頼者として連結**できる対応表を持った | 相談→受任のライフサイクルを跨いだ紐付け |
| 7 | **旧社名・グループ別名・主事件マーカーを命名に明示した** 例: `(旧サーモネット)` `中岡崇(HIRAKU)` `★★★(主事件)` [実績] | 別名でも現案件に寄せられる。別名辞書の「種」になる | 法人の旧称・グループ問題の解消 |
| 8 | **郵便物を最もクリーンなアンカーに据えた**（PostalMatter）[実績] | 差出人・宛名が明確で誤りが少ない。**正解データの起点**にできる | 他チャネルの教師データ源 |

**1行で言うと**: 「①正解を固定（SF ID埋め込み）」「②キーを分けて照合」「③精度を測って低い自動化は止める」「④ノイズを隔離」――この4つが精度向上の背骨だった。

## 2. 過去メールの紐付け vs これから入るメールの紐付け

同じ「メール紐付け」でも、**過去分（バックフィル）と新規分（リアルタイム）は考慮要素が違う**。仕組みは両方を別モードとして設計したほうが手戻りが少ない。

| 観点 | 過去メール（バックフィル） | これから入るメール（リアルタイム） |
|---|---|---|
| 案件の存在 | **既に存在**（終了済みも多い） | **まだ無い**ことがある（新規相談） |
| 文脈 | スレッド完結・確定済み | 未完・初動の1通だけ |
| 使える強シグナル | 事件番号・確定済み相手方・全文 | 差出人ドメイン・件名・過去の取引履歴 |
| 処理方式 | バッチ照合＋クラスタリング＋人手レビュー | **即時サジェスト**＋確信度しきい値 |
| 精度の効き所 | まとめて正解を固定（量で効かせる） | 1通の判定品質＋確定後フィードバック |
| 固有の難所 | 大量・重複・別案件移行の履歴 | **紐付け先が無い時の新規起票判定** |

**設計上の含意 [設計提案]**
- 過去分は「一度きりの大掃除」。**確定した結果を新規分の教師データに回す**設計にすると、リアルタイム精度が立ち上がりから高くなる。
- 新規分は「紐付け先が無い＝新規相談の可能性」を**起票トリガー**として扱う。ここを決めておかないと、未分類メールが溜まって手戻りになる。

## 3. チャネル別の考慮要素（メール / FAX / 郵便 / 電話 / 打ち合わせ）

チャネルごとに「使える強いシグナル」「弱点」「クリーンさ」が違う。**共通の背骨は `ALO_DerivedEvent__c`（source_type＋発生時刻でイベントを統合）**で、チャネル差はその手前の「どのキーで案件を当てるか」に出る。

| チャネル | 強いシグナル | 弱点・注意 | クリーンさ | 正規化の勘所 |
|---|---|---|---|---|
| **メール** | 差出人/宛先ドメイン、件名、スレッド継承、署名 | 転送・CC氾濫で当事者が薄まる／フリーメール | 中 | ドメイン→取引先辞書、署名から会社名抽出 |
| **FAX** | 送信元番号、表紙の宛先・件名 | **OCR必須**、手書き・画質、番号非通知 | 低〜中 | 番号→取引先辞書、表紙OCRの定型枠抽出 |
| **郵便** | 差出人・宛名（明確）、書面内の**事件番号**、裁判所からの送達 | 物理→スキャン取込の遅延 | **高（最クリーン）** | 事件番号 `court_case_key` で直結。PostalMatterを正解源に |
| **電話** | 着信/発信番号、fondesk着信メモ、Dialpad→Notta文字起こし | 要約品質、番号→人物の同定 | 中 | 番号→Contact辞書、文字起こしから依頼者名抽出 |
| **打ち合わせ** | GCal予定の出席者メール・タイトル・場所、Notta議事録 | タイトルが曖昧、内部MTGの混入 | 中 | 出席者メール→関係者照合、内部MTGの除外ルール |

**読み取れること**
- **「番号→人」「ドメイン→法人」の対応辞書**が、メール以外（FAX・電話）の精度を決める共通部品。**先に1つの辞書として作る**と全チャネルで効く [設計提案]。
- **郵便（特に事件番号入り書面）と打合せ出席者は精度が高い** → これらで確定した案件を、曖昧なメール・FAXの**当てに行く基準（教師データ）**に使う。
- チャネルが増えても背骨（DerivedEvent）は1本。**「源泉(source_type)」を必ず記録**しておくと、後からチャネル別の精度測定ができる。

## 4. 精度を「下げる」要因（先に潰しておくべき手戻り源）

仕組みを作る前に、この一覧を「既知の落とし穴」として共有しておく。

| 要因 | 症状 | 対策の方向 |
|---|---|---|
| 表記ゆれ（敬称/旧姓/法人格/全半角） | 同一人物が別人に見える | キー生成時に正規化（敬称・(株)・空白・全半角除去） |
| 同一依頼者・複数案件 | 依頼者一致だけでは案件が割れる | 依頼者一致は「候補母集合」止まり。事件名/相手方で二段確認 |
| 相談止まり／別案件移行 | 受任が無い・移行済みなのに寄せてしまう | 受任ペア無しは受任に寄せない。移行フラグを尊重 |
| 非案件クラスタ（研修・所内・メルマガ） | 大量メールで案件と誤認 | 研修/活動オブジェクトと `info@`等を除外 |
| フリーメール・共有アドレス | ドメインが効かない | 人名・件名で補強、共有アドレスは弱シグナル扱い |
| 番号非通知・OCR誤読（FAX/電話） | キーが取れない/誤る | 表紙定型枠の抽出、確信度を下げて人に回す |
| 自動分類の過信 | 誤った分野が量産される | **65.9%の教訓**＝低精度の自動書込みは禁止、サジェスト止まり |
| 浮遊リレーション | ノイズが母集合を汚す | **96.8%の教訓**＝浮遊データは隔離してから照合 |

## 5. 仕組みに先に組み込むべき「土台」要素（手戻り防止チェックリスト）[設計提案]

久保さんが実装に入る前に**決めておくと手戻りが激減する**8項目。

1. **正規化ルール（キー生成）の共通化** ― 依頼者キー・事件キーの作り方を1か所に定義。全チャネル共通。
2. **別名辞書を独立データに** ― 旧社名・グループ名・通称・代表者個人名↔法人。確定のたびに追記して育てる。
3. **番号/ドメイン→人・法人の対応辞書** ― FAX番号・電話番号・メールドメインを取引先/関係者に紐づける共通部品。
4. **確信度スコアと3段しきい値** ― `自動紐付け / 「これですか？」と確認 / 保留（未分類）`。閾値は後から調整できるパラメータに。
5. **人の確定・訂正をフィードバックする経路** ― 確定結果＝教師データ。採用キーの重み↑、却下パターンの抑制。
6. **source_type を持つイベント背骨**（DerivedEvent）に全チャネル集約 ― 後からチャネル別に精度を測れる形にしておく。
7. **「紐付け先が無い」時の新規起票トリガー** ― 新規相談メール等を取りこぼさない。
8. **監査ログと定点観測** ― 誰が/いつ/何を確定・訂正したかを残し、**チャネル別ヒット率・誤紐付け率**を継続測定。精度が上がっているかを数字で言える状態に。

## 6. まとめ（久保さんへの引き継ぎ骨子）

1. これまで効いたのは **「正解を固定（SF ID埋め込み）」「キーを分けて照合」「低精度の自動化は止める」「ノイズを隔離」** の4本柱。
2. **過去メールと新規メールは別モード**。過去分の確定結果を新規分の教師データに回すのが効率的。
3. **チャネルで強いシグナルが違う**が、背骨（DerivedEvent）と「番号/ドメイン→人」辞書は共通部品。郵便・打合せの高精度データを教師に、メール・FAXを当てに行く。
4. 手戻りを減らす鍵は **§5の8項目を先に決めること**。とくに「正規化ルール」「別名辞書」「確信度しきい値」「フィードバック経路」は、後から変えると影響範囲が大きいので最初に固めたい。
