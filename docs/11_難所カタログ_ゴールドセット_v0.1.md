# 難所カタログ / ゴールドセット v0.1（AIフィット向け・匿名化）

- 作成日: 2026-06-07
- 用途: **ALOコネクト（AIフィットが構築）の sf_case_id 解決**の、特徴量設計・confidence較正・回帰テストに使う**ラベル付き実例集**。
- 出自: 本プロジェクトで実データ・実メールから**検証済み**の難所を匿名化して類型化。**ALOしか作れないドメイン資産**。
- データ本体: [`data/goldset_anonymized.csv`](../data/goldset_anonymized.csv)（**36件**、機械可読。GPT監査CASELINKGOLD反映で expected-output スキーマ＋10タグ追加）。
- 守秘: **匿名化版**。実依頼者名・相手方名は役割記号（依頼者P/法人B/弁護士A〜C 等）。実データ接続は秘密保持＋権限境界の中で。

## 0. 使い方（AIフィットへ）

各エントリは「**入ってくるシグナル → 正解の紐付け先 → 根拠ルール → confidence想定 → 落とし穴**」の5点セット。

- **特徴量設計**: `link_basis`（根拠）が、resolver が持つべき特徴量・ルールの一覧になる。
- **confidence較正**: `confidence`（auto/review/unmatched）が、3段しきい値（自動確定/人レビュー/未分類）の目安。
- **回帰テスト**: 各行を `eval_set` に載せ、resolver 改修時の**デグレ検知**に使う。
- **ラベル**: `difficulty_tag` が難所タクソノミー。新しい難所が出たら追記して育てる。

> 重要原則（GPT監査でも確認済み）: これらは**確定ルールでなく強い候補特徴量**。`review` 行は自動確定せず人レビューへ。`auto` でも誤紐付けの**巻戻し（append-only link_decision_log）**を必ず持つ。

## 2. 代表エントリ（抜粋・読み物版）

### G-01 旧姓立件（離婚）★最重要
- **シグナル**: 件名「離婚の件」。依頼者は婚氏（依頼者P）、相手方は元配偶者（同姓）。
- **正解**: 受任（離婚）。ただし **SF側は旧姓（P'）で立件**されている。
- **根拠**: 離婚で旧姓に復するため旧姓で立件する実務がある。婚氏では繋がらない → **相手方氏＋分野（離婚）＋同時期**で同定、名寄せ台帳（婚氏↔旧姓）。
- **落とし穴**: 婚氏で検索して0件→未分類。**正規化では別姓で永久に繋がらない**。

### G-08 相手方代理人からのメール
- **シグナル**: 差出人が相手方の弁護士事務所ドメイン。
- **正解**: **自分（ALO）の当該受任案件**（相手方代理人とのやり取りも自案件の活動）。
- **落とし穴**: 「相手方＝別組織だから非案件」と誤判定する（一般ベンダーが最も誤る）。

### G-09 自所が手続主体（破産管財人）
- **シグナル**: 「破産管財人 ○○」宛、裁判所/債権者から。
- **正解**: 管財事件（**ALOが管財人＝手続主体**であって依頼者ではない）。
- **落とし穴**: 差出/宛先に自所名があると「内部メール」と誤判定して除外。

### G-12 利益相反確認の消去法（担当推定）
- **シグナル**: 所内 Chatter「@弁護士A @弁護士B 利益相反確認」。
- **正解**: 担当は**残りの弁護士C**（`assignee_prior`）。
- **根拠**: 利益相反確認は**担当していない弁護士に問い合わせる**運用 → 消去法。**単独で確定せず**物理フォルダ所在等で裏取り。
- **落とし穴**: @された弁護士を担当と誤認（メンション＝担当ではない）。

### G-18 SF未起票（Pattern E・規模約30%）
- **シグナル**: Boxに案件フォルダ実体あり、SFに受任レコード無し。
- **正解**: 未起票→事務局起票候補。
- **落とし穴**: SF JOINで欠落し「該当なし」で捨てる（**約30%が見えなくなる**）。

### G-26 添付↔Gmail identity bridge（既知ブロッカー）
- **シグナル**: メール添付の自動保存。`msg_key` と `gmail id` が交差しない。
- **正解**: **保留**（bridge構築まで自動保存禁止）。
- **根拠**: `DD-EMAILATTACH-001` の既知ブロッカー（identity bridge未解決、conflict_risk 1,592）。
- **落とし穴**: 添付を誤案件/重複保存。AIフィットは**この未解決前提を知っておく必要**がある。

## 3.5 GPT監査（CASELINKGOLD = PASS_WITH_NOTES）反映

ベンダー渡し前の必須修正として、GPTの指摘を反映済み：

### (a) auto指定を弱めた（誤紐付け防止）
**純粋autoは廃止。** 危険な3つは「条件付きauto（`auto_if_condition` を満たす時のみ）」へ降格：
- **G-08 相手方代理人** → review。auto は「事件番号 or 相手方名 ＋ thread継続 ＋ 直近案件が一意」の時のみ。同一代理人は複数案件のrepeat playerになるため。
- **G-20 事件番号** → 「裁判所＋元号年＋事件種別＋番号＋matter側文脈」が揃う時のみauto。番号単独は別裁判所・派生事件（控訴/保全/執行）で衝突。
- **G-03 中黒正規化** → 候補生成キーであって同一性確定キーではない。法人番号/住所/代表者/ドメイン/SF既存関係の一致時のみauto。
- G-14/G-23 は概ねauto可だが**例外条件**（講師依頼が顧問に紐づく／業務system通知が案件リンクを含む）を `forbidden_signals` に明記。

### (b) goldset に expected-output スキーマを追加（resolver のテスト用）
CSVに以下の列を追加：`expected_action`（link_to_matter/link_as_role/flag_non_matter/create_matter_candidate/merge_party/split/hold/human_review）、`expected_object_type`、`confidence_band`、`auto_allowed`(yes/conditional/no)、`auto_if_condition`、`allowed_signals`（使ってよい）、`forbidden_signals`（使ってはいけない）、`must_not_link_to`、`risk_type`（confidentiality/billing/workflow_risk）。

### (c) auto/review/unmatched の機械的定義（実装者向け）
- **auto**: 一意性を示す**独立シグナル2系統以上**があり、`must_not_link_to`/`forbidden_signals` 条件に当たらない。
- **review**: 候補はあるが、競合・古い名称・複数案件・手続主体・CCのみ等の事情がある。
- **unmatched**: 紐付け禁止または保留（identity bridge未解決等）。

### (d) SF未起票（G-18）の理由分割
`no_sf_match_reason = new_matter_candidate / non_matter / insufficient_signal / duplicate_suspected`。SF無し全件を一律起票候補にしない。

### (e) 難所10タグ追加（G-27〜G-36）
共同受任・復代理 / 保険・共済・保証会社 / 破産再生の債権者多数 / 家事民事刑事の混在 / 派生事件(控訴抗告保全執行) / 外国当事者・翻訳名 / 顧問先グループ会社 / 旧担当者・退職者メール / 裁判所・管財人・監督委員の通知 / 一括メール・CCだけ関与。

## 5. `_alo_dir.json` 構造リファレンス（正解ラベル一括生成の素材・実データ検証済み）

各案件フォルダの `_alo_dir.json`（schema `alo-dir-map-v2`）は、**フォルダ→SF確定 ＋ サブ構造 ＋ AI処理指示**を持つ。resolver はこれを**正解ラベル・除外判定・保存先slot**にそのまま使える。

**トップレベル**: `sf`（SF Record ID＝確定アンカー）/ `alo_matter_uri` / `case` / `case_root_box` / `case_summary_box_file_id`（AI生成の案件サマリ template `ALO-CS-1.0`）/ `manifest_status`

**`subs[]` の主要フィールド**:
| フィールド | 値域 | resolver での使い道 |
|---|---|---|
| `cat` | 通信_受 / 通信_送 / 文書データ / 預かり資料 / 会計 / 訴訟記録 / 書面 / 債権執行 / 破産申立 / **管財** / 不動産処分 / 契約書 / 請求書明細 / 関係者共有 / 書式 / 作業フォルダ / アーカイブ / 未分類 / その他 | フォルダ分類の統制語彙。保存先slot |
| `dir` | incoming / outgoing / mixed | **受信メール→通信_受 / 送信メール→通信_送** |
| `ai` | normal / **deprioritize / exclude** | deprioritize/exclude は**当事者メール対象外** |
| `profile` | sub / **sys** | sys は内部用（old・作業フォルダ・BROMIUM）→対象外 |
| `status` | archive / working / deletion_candidate | 非対象/除外 |
| `worker` | 武市 等 | 内部作業者フォルダ＝対象外 |

**実データで確認した裏づけ（匿名化）**:
- **自所が手続主体（G-09/10）**: 破産管財事件フォルダに `cat=管財`「管財人業務／不動産処分」が実在 → ALOが管財人。
- **顧問契約（a04）＝顧問レイヤー（G-15/33）**: 顧問契約フォルダ sf 接頭辞 **a04**、`顧問料請求書（毎月メール送信要）`、グループ会社サブ（複数法人）が実在。
- **内部作業/アーカイブ除外（G-37）**: `武市作業`/`宮澤作業`/`old`/`BROMIUM` は `ai:deprioritize|exclude`・`profile:sys` → 当事者通信と混同しない。

> **一括ラベル生成の手順（読み取りのみ）**: DD-BOXCATALOG-001 で案件↔ファイルを網羅取得 → 各フォルダの `_alo_dir.json` で `sf`（確定）・`cat`/`dir`/`ai`/`profile` を付与 → Gmailラベル `ALO/<sf>` と突合 → 匿名化して `eval_set` 行に展開。**`ai/profile/status` で非対象フォルダを最初に除外**するのが精度の肝。
