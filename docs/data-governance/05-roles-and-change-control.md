# 05. 権限と変更管理

## 役割

| 役割 | できること |
| ---- | ---------- |
| **Ingestor（投入者）** | landing への投入、検査関数の実行、dirty ラベル付け。 |
| **Reviewer（レビュア）** | 昇格 PR のレビュー。チェックリスト完了を確認し承認/差し戻し。 |
| **Promoter（昇格者）** | 承認済み PR をマージし、prod マイグレーションを適用。 |

小規模運用では1人が兼ねても構いませんが、**昇格 PR は必ずレビュー手順（チェックリスト
確認）を踏む**こと。セルフレビューでも、チェックリストを埋めずにマージしないこと。

## 変更管理の鉄則

1. **prod への変更は必ず PR 経由。** DB を直接いじって済ませない。
2. **DB は再構築可能に保つ。** すべての構造・参照データは `supabase/` 配下の
   マイグレーションに表現され、ゼロから再現できる状態を維持する。
3. **手で直したら、必ずマイグレーションに反映する。** さもないと DB が「一人歩き」し、
   Git と実体が乖離する。乖離は事故の温床。
4. **削除・上書きは慎重に。** prod のデータ修正は、対象を確認し、`version` を上げる形で
   履歴を残す。物理削除はレビュー必須。

## 認証情報の分離

- `codex-staging` と `codex-prod` は別プロジェクト＝別の接続情報・別の API キー。
- prod の `service_role` キーは厳重に管理し、CI やローカルの自動投入には使わない
  （prod 書込はレビュー済みマイグレーション適用時のみ）。
- キー類はリポジトリにコミットしない（CI の secrets / 環境変数で管理）。

## 監査

- すべての行が来歴メタデータ（`source` / `ingested_by` / `validated_by` / `validated_at`）
  を持つため、「いつ・誰が・どの経路で」入れたかを後から追える。
- 昇格は Git 履歴（PR）に残るため、prod の各データがどの PR で入ったか追跡できる。
