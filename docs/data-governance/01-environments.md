# 01. 環境の定義

## 2プロジェクト構成

テスト用と本番用を **別の Supabase プロジェクト** に分離します。別DB・別認証情報なので、
テストデータが本番に混入する事故が物理的に起こりません。

| プロジェクト | 役割 | 含むスキーマ |
| ------------ | ---- | ------------ |
| `codex-staging` | プレ本番・試作・テスト | `codex`(共通), `landing`, `staging` |
| `codex-prod` | 本番・公開 | `codex`(共通), `prod` |

> 既存プロジェクト `alo-connect` はアプリ用途と推測されるため、本用途には使いません。
> `codex-staging` / `codex-prod` は新規作成、または `asai-dot's Project` を staging に
> 割り当てる等の対応を provisioning 時に確定します。

## スキーマの役割と書込可否

| スキーマ | プロジェクト | 役割 | 書込 | 削除/更新 |
| -------- | ------------ | ---- | ---- | --------- |
| `landing` | staging | 検疫区画。生データ・dirty可（要ラベル） | 自由 | 自由 |
| `staging` | staging | 検査通過した昇格候補 | 検査関数経由のみ | レビュー前提 |
| `prod` | prod | 公開リファレンス | **昇格スクリプト経由のみ** | レビュー必須 |
| `codex` | 両方 | 共通の enum・関数・メタ列ヘルパ | マイグレーションのみ | マイグレーションのみ |

## 「これ書き換えていい？」早見表

- **landing のデータ** → 自由に壊して良い。使い捨て前提。
- **staging のデータ** → 検査関数で作り直すのが原則。手で直さない。
- **prod のデータ** → ✗ 直接書き換え禁止。必ず PR レビュー済みの昇格・修正マイグレーション経由。

## アクセス権（prod のロックダウン）

`codex-prod` では「携帯から OK 連打で生データが直接入る」経路を物理的に塞ぎます。

- `anon` / `authenticated` ロールには `prod` スキーマへの `INSERT/UPDATE/DELETE` を付与しない（読取のみ、必要なら）。
- 書込は `service_role` ＋ レビュー済みマイグレーション経由に限定。
- RLS を有効化し、書込ポリシーは付けない（=アプリ経由の書込を拒否）。

詳細な GRANT/RLS は `supabase/prod/migrations/` を参照。

## 命名規則

- スキーマ: 小文字スネークケース（`landing` / `staging` / `prod`）。
- テーブル: 単数形・スネークケース（`municipality`, `prefecture`）。
- 自然キー列: 業務上の正式名（`muni_code` 等）。サロゲートキーは `id`(uuid)。
- マイグレーション: `NNNN_説明.sql`（連番・前ゼロ4桁）。
