# 04. 汚いデータ（dirty）の取扱い規程

綺麗なデータベースを守る前提は「綺麗なデータしか入れない」ことです。しかし現実には、
作業の都合で汚いデータを一時的に置きたい場面があります。その場合の唯一の許容ルールが
本規程です。**汚いデータは「汚い」と明示ラベルした landing にのみ置けます。**

## 原則

1. **無印（ラベルなし）の汚いデータは存在してはならない。** 入れるなら必ずラベルする。
2. **dirty は landing 限定。** staging / prod へは絶対に昇格しない（CHECK 制約で保証）。
3. **dirty には理由を残す。** `notes` に「なぜ汚いか／どう直すべきか」を書く。

## ラベルの付け方

| 状況 | quality_status | 補足 |
| ---- | -------------- | ---- |
| 未検査で投入しただけ | `unverified` | 既定値。検査を待っている状態。 |
| 検査で弾かれた | `quarantined` | 検査関数が自動付与。原因を確認する。 |
| 汚いと分かっていて意図的に置く | `dirty` | 人が明示的に付与。`notes` 必須。 |
| 検査通過 | `clean` | 昇格可能。 |

## dirty を置いてよい典型例

- 取得元が既に汚いことが分かっており、クレンジング前の原本を保全したい。
- 検査ルールそのものを検討するために、悪い例を手元に残したい。
- 部分的に欠損していて、後で補完予定のデータ。

いずれも **landing に置き、`dirty` ラベルと理由を付ける**ことが条件です。

## dirty の出口

dirty 行は次のいずれかで処理します。放置しません。

- **修正** → クレンジングして `unverified` に戻し、再検査。
- **破棄** → landing から削除。
- **保全** → 原本として残す場合も `dirty` のまま。昇格対象には決してならない。

## やってはいけないこと

- prod / staging に汚いデータを「とりあえず」入れる。
- ラベルなしで汚いデータを混ぜる（どれが綺麗か識別不能になる）。
- 携帯等から検査を飛ばして OK 連打で投入する（landing 直投入でも検査は必須）。
