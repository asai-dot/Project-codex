# 案件紐付けロジックの言語化 ―「Salesforceのこれですか？」サジェスト設計

- 作成日: 2026-06-05
- 用途: 久保さんへの進捗説明資料（判断過程の言語化編）
- 目的: メール・郵便物・ファイルを Salesforce 案件に紐付けるとき、**我々が実際にどう判断しているか**を言語化し、案件画面に出す「Salesforceのこれですか？」サジェストの根拠ルールにする。

> このドキュメントは、これまでの紐付け作業（Gmail ラベリング、TASK-A 案件名特定、TASK-B 相談→受任反映、Box フォルダ整理）の試行錯誤から抽出した**当事務所固有の判断知見**をまとめたものです。汎用的な名寄せ理論ではなく、「浅井法律事務所の業務特性・データの並び方」に沿った実務ルールです。

---

## 0. 基本思想 ― なぜ機械的に紐付くのか

紐付けが成立する根拠は、当事務所のデータが以下の3点で**規則的に並んでいる**ことにあります。

1. **背骨は `sf_record_id`**。Salesforce の1レコード（受任案件・相談）が1案件の単位。
2. **命名規則が3システムで共通**。
   - Gmail ラベル: `ALO/<SF ID>_依頼者＿事件名`
   - Box フォルダ: `五十音_依頼者_事件名`
   - 区切り文字「＿/_」の左が**依頼者**、右が**事件名**。
3. **Box フォルダ分類 ＝ Salesforce オブジェクト分類**（受任案件/相談/顧問契約/顧問相談/研修）。

→ つまり「依頼者名」と「事件名」という2つのキーで、メール・フォルダ・SFレコードを突き合わせられる。サジェストはこの突合のスコアリングです。

## 1. 紐付け判断の優先順位（サジェストのランキング・ロジック）

新しいメール／郵便物が来たとき、どの案件かを**この順に**判定します。上位ほど確実で、上位で確定すれば下位は見ない。

### Tier 0 ― 明示的な SF ID 参照 〔信頼度: 確定〕
- メール／スレッドに既に `ALO/<SF ID>_…` ラベルが付いている、または本文・件名に SF ID・案件管理番号がある。
- → **即確定**。スレッド全体に継承（Tier 5）。

### Tier 1 ― 「依頼者名 ＋ 事件名」の一致 〔信頼度: 高〕
- 件名・本文・差出人名に、SFレコードの **依頼者名（Account名）** と **事件名** の両方が現れる。
- 例: 件名「久保浩二様 交通事故の件」→ `久保浩二_損害賠償請求事件(交通事故)` に高スコア。
- 依頼者名のみ一致は中スコア（同一依頼者が複数案件を持つため。→ §3-3）。

### Tier 2 ― 関係者グラフによる一致 〔信頼度: 高〜中〕
- 差出人／宛先／本文の人名・社名が、その案件の **相手方（OpponentParty）・関係者（RelatedContact）・代理人** に一致する。
- 当事務所は相手方代理人とのメールが多く、**「相手方名」での一致は依頼者名と同等に強い**シグナル。
- 例: 相手方「広和建設」→ `隈元建設_売掛代金請求事件(vs広和建設)`。

### Tier 3 ― メールアドレス／ドメインによる一致 〔信頼度: 中〕
- 差出人ドメインを Account/Contact のメールドメインに突合。
  - 顧問先・法人クライアントの独自ドメイン → その取引先に強く結びつく。
  - 相手方事務所のドメイン → 相手方経由で案件特定。
- フリーメール（gmail等）は弱いので、人名・件名（Tier 1/2）で補強。

### Tier 4 ― 事件番号・裁判所による一致 〔信頼度: 高（訴訟案件）〕
- 「令和○年(ワ)第○号」等の事件番号、裁判所名を `JurisdictionCourt__c` の `court_case_key` に突合。
- 訴訟段階の案件では**最も安定したアンカー**。手続が交渉→調停→訴訟と進んでも事件番号で時間軸を固定できる。

### Tier 5 ― スレッド継承 〔信頼度: 継承元に準拠〕
- スレッド内の1通でも案件が確定すれば、**スレッド全体を同一案件に**継承。
- 返信・転送で件名が変わっても継承を維持。

### Tier 6 ― 時間的近接 ＋ 担当弁護士 〔信頼度: 補助〕
- 直近活動日が近い案件、担当弁護士（浅井T/武市T/宮澤T）の案件を優先候補に。
- 単独では確定にせず、上位ティアの**同点候補の絞り込み**に使う。

### Tier 7 ― 相談⇄受任の昇格関係 〔信頼度: 中〕
- 相談（`a0D`）が受任（`a0A`）に昇格している場合、**昇格前のメールは相談に、昇格後は受任に**割り当てつつ、同一依頼者として束ねる。
- 名称が変わるため別案件に見えるが、TASK-B のペア表で結びつける（→ §3-4）。

## 2. 当事務所特有の「効かせどころ」（業務特性ベースの知見）

汎用ロジックに、当所の業務・データの癖を重ねることで精度が上がります。

| # | 当所の特性 | 紐付けへの効かせ方 |
|---|---|---|
| A | **`依頼者＿事件名` の区切り規則** | 「＿」「_」で機械分割し、左＝依頼者キー／右＝事件キーとして別々に照合 |
| B | **五十音プレフィックス**（あ_/い_） | フォルダ突合時のソート・名寄せ補助。読み仮名の手掛かり |
| C | **法人の旧称・別名・グループ名** | 「旧サーモネット」「中岡崇(HIRAKU)」のように別名を併記。別名辞書を持ち、旧称メールも現案件に寄せる |
| D | **同一依頼者の複数案件分岐** | 依頼者名一致だけで確定しない。事件名・相手方・時期で必ず二段確認（→ §3-3） |
| E | **相談→受任で事件名が変わる** | 「法律相談」→「○○請求事件」。名称ズレを許容し、相談・受任ペア表で連結（→ §3-4） |
| F | **研修・対外活動は「案件」ではない** | 大量メールが溜まっても案件化しない。`a1s`/研修区分は案件サジェストから除外し「活動」バケットへ |
| G | **交通事故(LAC)など定型カテゴリ** | 「(LAC)」「交通事故」などの定型語で分野を即時推定し候補を絞る |
| H | **顧問先の recurring メール** | 顧問契約先からの定例連絡は顧問オブジェクトに寄せ、個別受任案件と混ぜない |
| I | **郵便物が最もクリーンなアンカー** | `PostalMatter__c` は宛先・差出人が明確で誤りが少ない。郵便物で確定した案件をメール紐付けの教師データに使う |

## 3. 判断の難所と回避ルール（誤紐付けの典型）

実際の作業で繰り返し迷った／間違えやすいパターンと、その回避ルールです。**ここが「Salesforceのこれですか？」と人に確認を促すべき場面**です。

### 3-1. 取引先(Account)レベル vs 案件(Matter)レベルの取り違え
- 症状: メールが「会社全体」宛で、特定案件に絞れない。
- 例: `株式会社ミックス`（取引先=001）配下に複数案件。
- **回避**: 依頼者一致のみのときは Account にいったん寄せ、事件名/相手方が出るまで Matter を確定しない。

### 3-2. 主事件 / 付随・個人事件の分岐
- 例: `★★★(主事件)合同会社HIRAKU_破産申立事件` と `HIRAKUホールディングス_破産申立事件`、`中岡崇(HIRAKU)_債務整理`。
- 例: `田中外士夫商店` に「法人破産」「池本勇輝個人」「池本昭夫個人」の3案件。
- **回避**: 「(主事件)」「(個人)」「(法人)」マーカーを優先キーにし、人名（代表者個人 vs 法人）で振り分け。

### 3-3. 同一依頼者・複数案件
- 例: `龍村美術織物` が「売掛金回収」と「対エステックサービス」の2件。`福永孝文` が受任「遺産確認」と相談「法律相談」。
- **回避**: 依頼者一致は**候補の母集合**に過ぎない。事件名・相手方・時期の二段目照合を必須にする。

### 3-4. 相談止まり / 別案件への移行
- 例: `アダチ`＝相談のみ（受任なし）。`別案件移行済` フォルダの存在。
- **回避**: 相談レコードに受任ペアが無ければ受任案件へ寄せない。移行済みフラグがあれば移行先に寄せる。

### 3-5. 表記ゆれ・敬称・旧姓
- 例: 「様」付き／なし、旧姓、法人格（株式会社/有限会社/(株)）の有無。
- **回避**: 正規化（敬称・法人格・空白・全半角を除去）したキーで照合。元表記も保持。

### 3-6. 非案件クラスタ（研修・活動・所内業務）
- 例: 北海道包括外部監査人（メール1,131件）、AIシステム導入検討。
- **回避**: 件数の多さに引きずられない。研修・対外・所内業務は案件サジェスト対象外。

## 4. サジェスト提示時の運用ルール（UI挙動）

スコアリング結果を、案件画面でどう見せるかの方針です。

| 信頼度 | 条件（目安） | 画面挙動 |
|---|---|---|
| **High** | Tier 0/1/4 が一致、または依頼者＋事件名＋相手方が揃う | 自動で当該案件に紐付け（バッジ表示、ワンクリックで解除可） |
| **Mid** | 依頼者一致＋分野推定はあるが事件特定が弱い | **「Salesforceのこれですか？ → ①案件A ②案件B」と候補提示**。人が1クリックで確定 |
| **Low** | 名寄せ候補が弱い／非案件の可能性 | 紐付けず「未分類」。候補を薄く提示するに留める |

### フィードバック・ループ（学習）
- 人が確定／訂正した結果を**教師データ**として蓄積。
  - 採用された候補 → そのキー（ドメイン・別名・相手方名）の重みを上げる。
  - 却下された候補 → 同パターンの誤りを抑制。
- 特に **郵便物（最クリーン）と相談→受任ペア表（TASK-B）** を正解データの起点にする。
- 別名辞書（旧社名・グループ名）は確定のたびに追記し、Cの精度を継続的に上げる。

## 5. まとめ（久保さんへの説明骨子）

1. 紐付けは「`sf_record_id` を背骨」に、**依頼者名・事件名の共通命名規則**でメール／フォルダ／SF案件を突き合わせる仕組み。
2. 判断は **7段階の優先順位**（明示ID → 依頼者+事件名 → 関係者 → ドメイン → 事件番号 → スレッド継承 → 時間/担当 → 相談⇄受任）で行う。
3. 当所固有の癖（旧社名・グループ別名・同一依頼者の複数案件・相談→受任の名称変化・研修等の非案件）を**ルール化**し、機械が迷う場面だけ「Salesforceのこれですか？」と人に確認する。
4. 人の確定結果を学習に回し、別名辞書とスコア重みを育てることで、サジェスト精度が継続的に上がる。
