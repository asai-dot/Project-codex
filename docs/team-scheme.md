## 0. プロジェクトの目的（30秒でわかる版）

日本の法令（e-Gov 法令データ）から「**法令上の定義語**（statutory definitions）」を
抽出・正規化・品質保証し、機械可読なデータセット（JSONL）として整備するプロジェクト。

成果物の最小単位は JSONL の1行 ＝ 1定義レコード。スキーマは §5 を参照。

> 例:
> ```json
> {"term": "後見人", "definition": "未成年後見人及び成年後見人をいう。", "law_id": "129AC0000000089", "law_name": "民法", "article": "10", "item": null, "uri": "egov:129AC0000000089:art:10", "scheme": "jp_statutory_definition", "authority_rank": 100, "source": "egov", "definition_type": "paren_definition", "confidence": "high"}
> ```

私たちが守るべき価値は **正確さ ＞ 網羅性 ＞ 速度**。法令データは「それっぽい嘘」を
1件出すだけで信頼を失う。迷ったら confidence を下げ、根拠（uri）を残す。

## 2. ロール定義

### 2.1 クロードヘッド（Claude Head）— 司令塔
- **担当**: 要件の理解、タスク分解、設計判断、各エージェントへの割り当て、成果の統合、
  最終レビュー、Git/PR の統括。
- **権限**: 設計の最終決定権を持つ唯一のAI。仕様変更・スキーマ変更を承認する。
- **アウトプット**: 「やること（仕様）・やらないこと（スコープ外）・受け入れ基準（DoD）」を
  明示したタスク指示。曖昧なまま下に流さない。
- **原則**: 自分で手を動かしすぎない。下位エージェントが迷わない粒度まで割って渡す。

### 2.2 ワーカーちゃん（Worker-chan）— 実装の主力（Claudeワーカー）
- **担当**: 文脈理解・判断を要する中〜大規模の実装、抽出パイプライン本体、リファクタ、
  複数ファイルにまたがる変更。
- **得意**: 「何を作るべきか」を補完しながら作れる。曖昧さに強い。
- **渡し方**: 設計意図＋制約を渡せば、細部は任せてよい。

### 2.3 ローカルちゃん（Local-chan）— 大量・前処理担当（ローカルモデル）
- **担当**: バルク処理、正規表現ベースの一次抽出、整形・クレンジング、重複排除、
  低コストで何度も回す反復処理。
- **得意**: ローカルで完結＝**機密/大容量データを外に出さずに回せる**。安価・高速。
- **渡し方**: 入力・出力・判定ルールを機械的に決め切って渡す。判断が要る作業は振らない。

### 2.4 コーデックスちゃん（Codex-chan）— 精緻な手（OpenAI Codex）★今回オンボード対象
- **担当**: 仕様が固まったタスクの**忠実で確実な実装**、単体テスト作成、エッジケース潰し、
  バグの局所修正、地道なパッチ、データ検証スクリプト。
- **性格づけ**: 賢く・真面目に・コツコツ。手戻りの少ない「精緻な手」。
- **得意**: スコープが明確な作業を、小さく確実な差分で仕上げる。テストで裏を取る。
- **渡し方**: 受け入れ基準とサンプル入出力をセットで渡す（§4 ハンドオフ様式）。
- 実務ルールの全文は [`AGENTS.md`](../AGENTS.md)。

### 2.5 GPT監査スキーム（GPT Audit）— 独立監査
- **担当**: 成果物の第三者レビュー。特に **データ品質・定義文の正確さ・confidence 判定の妥当性・
  ハルシネーション検出・スキーマ規約違反**のサンプリング検証。
- **独立性の原則**: **実装したエージェントと監査は必ず別系統**。自分の成果を自分で監査しない。
  コーデックスちゃんが作った差分は GPT監査が見る、という分離を保つ。
- **アウトプット**: 合否ではなく「指摘＋根拠＋推奨修正」。再現可能な指摘を出す。

## 3. 標準ワークフロー

```
[1] asai が要望を出す
        │
[2] クロードヘッド が要件化 → タスク分解 → 受け入れ基準(DoD)を定義
        │
[3] 割り当て:
      ・判断が要る実装        → ワーカーちゃん
      ・大量/機械的な前処理     → ローカルちゃん
      ・仕様確定済みの精緻な実装 → コーデックスちゃん
        │
[4] 各自が作業 → ブランチに小さくコミット
        │
[5] クロードヘッド が成果を統合・整合性チェック
        │
[6] GPT監査 が独立レビュー（品質・正確さ・規約）
        │
[7] 指摘を反映 → ヘッドが最終確認 → PR を更新/マージ可能化
```

- **手戻りを減らす鉄則**: 仕様の曖昧さは [3] より前で潰す。下流に行くほど修正コストが上がる。
- **詰まったら止める**: どのエージェントも、判断に迷ったら勝手に進めず、ヘッド（最終的には asai）に
  確認を上げる。「賢く真面目」とは、わからないことをわかると言えること。

## 4. ハンドオフ様式（タスクをエージェントに渡すときの型）

クロードヘッドが下位エージェントに渡すタスクは、必ず次の形にする。
特にコーデックスちゃん／ローカルちゃんには曖昧さを残さない。

```markdown
## タスク: <一行で目的>
- 担当: <コーデックスちゃん / ワーカーちゃん / ローカルちゃん>
- 背景: <なぜやるか / 関連ファイル>
- やること: <具体的な変更内容>
- やらないこと(スコープ外): <触ってはいけない範囲>
- 入力例 / 出力例: <サンプル。データ系は JSONL の前後を必ず添える>
- 受け入れ基準(DoD):
  - [ ] <テストが通る / スキーマ検証が通る 等、検証可能な条件>
- 監査ポイント: <GPT監査が重点的に見る観点>
```

## 5. データスキーマ（成果物の唯一の正）

JSONL 1行 = 1定義レコード。フィールドは固定。勝手に増減・改名しない（変更はヘッド承認制）。

| フィールド | 型 | 説明 | 例 |
|---|---|---|---|
| `term` | string | 定義される語 | `"後見人"` |
| `definition` | string | 定義文（原文準拠） | `"未成年後見人及び成年後見人をいう。"` |
| `law_id` | string | e-Gov 法令ID | `"129AC0000000089"` |
| `law_name` | string | 法令名 | `"民法"` |
| `article` | string | 条 | `"10"` |
| `item` | string\|null | 項/号（無ければ null） | `null` |
| `uri` | string | 出典URI（再現可能な根拠） | `"egov:129AC0000000089:art:10"` |
| `scheme` | string | スキーム識別子 | `"jp_statutory_definition"` |
| `authority_rank` | number | 典拠の格付け | `100` |
| `source` | string | 取得元 | `"egov"` |
| `definition_type` | string | 定義の型 | `"paren_definition"` |
| `confidence` | string | 確信度 | `"high"` |

**品質バー（最低ライン）**
- すべての行は **有効なJSON**で、上記キーを過不足なく持つ。
- `definition` は**原文に忠実**。要約・言い換え・創作は禁止（ハルシネーション厳禁）。
- 根拠 `uri` は必ず埋める。たどれない定義は出さない。
- 確信が持てない行は捨てるか `confidence` を下げる。**「正確さ ＞ 網羅性」**。

## 6. Git / ブランチ運用

- 各作業は専用ブランチで。命名: `claude/<topic>-<id>`（例: `claude/codex-setup-docs-z0dnc`）。
- コミットは**小さく・意味単位で**。メッセージは「何を・なぜ」を一行目に。
- 直接 `main` に push しない。指定ブランチへ push → **ドラフトPR**を作成 → レビュー後にマージ。
- 大容量データ（`*.jsonl` の生成物）は配置場所のルールに従う（生データは `data/egov/` 配下）。

## 7. やってよいこと / だめなこと（全員共通）

✅ よいこと
- 仕様が曖昧なら**作業を止めて確認**を上げる。
- 差分は小さく、テスト・検証で裏を取る。
- 出典（uri）を必ず残す。

🚫 だめなこと
- スコープ外のファイルを「ついでに」変更する。
- スキーマを無断で変える。
- 根拠のない定義を創作する／確信度を盛る。
- 自分の成果を自分で監査して通す（監査は独立系統で）。
