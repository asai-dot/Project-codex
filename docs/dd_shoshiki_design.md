## 1. 書式エントリのメタ情報モデル

1つの書式が持つべき情報の体系:

```
書式エントリ
 ├ 識別
 │   ├ 書式ID / タイトル
 │   └ 手続フロー上の位置 (手続名 + ステップ順序)
 ├ 使用文脈
 │   └ いつ / 誰が / 何のために作成するか
 ├ 根拠法令 (複数可)
 │   └ 実体法 + 手続法 の両方 (例: 会社法 + 商業登記法)
 ├ 床(記載): 法定記載事項   ← e-Gov 条文各号由来
 ├ 床(要件): 充足すべき要件
 │   ├ 法定要件    (条文が根拠。条文改正まで不変)
 │   └ 行政手引き  (手引き・通達が根拠。改訂で変わる)
 └ 床(添付): 添付書類チェックリスト
```

**要件のソース種別を分けて持つ理由**: 法定要件は条文が正本で不変。
行政手引き由来は更新頻度・権威が違い、運用の中で変わる。混在させると
どちらが変わったかが追えなくなる。

## 2. 定型化のしやすさスペクトル

「法定記載事項の遊びの少なさ」で書式ドメインを並べると:

```
← 遊びが多い (定型化しにくい)       遊びが少ない (定型化向き) →

契約書     訴状・準備書面   保全申立   執行申立   登記    許認可申請
(自由度高)  (主張の組立次第)  (要件限定)  (書式固定) (法定列挙) (法定列挙)
```

**保全・執行** が登記と並んで最右端に来る理由: 裁判所が書式を公式に指定
しているため「書式床」が存在し、記載ミスが却下に直結する。裁判所 PDF が
一次ソースになる点が登記(e-Gov)と異なる。

## 3. 床の多層構造 (ドメイン別)

### 登記(商業・不動産)
```
変更登記申請書
 ├ 実体床 (会社法 915①→911③): 何が変わったか
 └ 手続床 (商業登記法 17条等): 申請書の書き方・添付書類・登録免許税
```

### 許認可申請
```
許認可申請書 (例: 建設業許可)
 ├ 要件床 (建設業法 7条): 申請者が満たすべき許可基準
 ├ 記載床 (施行規則 2条): 申請書に書くべき事項
 └ 添付床 (同規則 2条・4条): 証明書類チェックリスト
         ↑ 添付書類一枚一枚が自身の床を背負う
```

### 保全・執行
```
仮差押命令申立書 (民事保全法)
 ├ 要件床 (保全法 20条): 被保全権利 + 保全の必要性 の2要素
 ├ 記載床 (保全規則)
 ├ 添付床: 疎明資料
 └ 書式床 (裁判所指定書式): 申立書テンプレート自体が床 ← 登記にない層
```

## 4. 書式 ↔ 文献のリンク構造

alias が「条文語 ↔ 実務語」の橋になり、書式 slot と文献 TOC 見出しを
自動で紐付ける:

```
条文語 (e-Gov)               alias               実務語(文献・書式)
─────────────────────────────────────────────────────────────────
「増加する資本金及び      ←→  「資本金の額」   ←→  変更登記申請 slot
 資本準備金に関する事項」      「資本金として計上」    文例33型 第4項
 (199①五)                                          実務書 p.254
```

**書式 → 本文**: 申立書の slot を埋める際に「この slot の解説は○○書 p.123」
**本文 → 書式**: 文献の節を読んでいるときに「この説明は○○申立書の slot に対応」

bib_toc の TOC 見出しと書式 slot の alias が重なる箇所から**参照マップ**が
半自動で作れる。本文未収録でも TOC レベルで「どの本の何章を見ればよいか」が出る。

## 5. 同一根拠の複数書式 → 比較マトリクス

同一法令を床に持つ書式が複数集まると、スロット構造を縦横にして
**差分マトリクス**が自動生成できる:

```
遺言書パターン (床: 民法 968〜973 条)

              自署要件  証人  公証人  日付  押印  財産特定
自筆証書       ◎必須    —     —      ◎    ◎    自由記載
公正証書       口述のみ  2名   ◎     ◎    ◎    厳格
秘密証書       封印のみ  2名   ◎     —    ◎    自由記載
危急時(口授)   —       3名   —      ◎    —    簡略可
```

**ニッチ書式の一発サーフェス**: 条件(証人1名しかいない危急時など)を
入力すると、当てはまるパターンが絞り込める。

## 6. OCR パイプラインへの含意

書式一覧・図表一覧を持つ本は**ページ種別フラグ**が要る:

```
本の中の特殊ページ種別
 ├ 目次       → 既存 bib_toc (章見出し+頁)
 ├ 書式一覧   → ★未取得: 書式番号+タイトル+頁  ← 構造化パーサーが必要
 ├ 図表一覧   → ★未取得: 図表番号+タイトル+頁
 ├ 判例索引   → ★未取得: 事件名/年月日+頁
 └ 事項索引   → ★未取得: キーワード+頁(逆引き)
```

書式一覧が取れると SOURCES.md に手書きしていた書誌情報が自動で構造データになる。
OCR パイプラインに**ページ分類器**を一段挟み、書式一覧らしいページ
(連番+書式番号+頁番号のパターン)を検出して専用パーサーに回す。

## 7. デジタル庁オンライン化との将来接続

行政手続きのデジタルフォームは構造的に「フォームフィールド = 記載床」。
e-Gov 電子申請フォームが整備されてくると:

```
自社書式 slot  ←→  e-Gov フォーム field  (照合・マッピング)
```

許認可申請のルーティン部分(書類作成・提出)は代替されうる。
残る価値は:
- 要件充足の**判断** (この申請者は要件を満たすか)
- 複数許認可の**組み合わせ**
- 行政との**折衝・補正対応**
- 手引きに書いていない**実務慣行**

「書式の作成・提出」→「その書式を使うべきかの判断と前捌き」へ価値が移行。
