## G-01. 業務レコードの脊椎（背骨ID） [D-001 / P0:Q-003]

- **裁定**: 相談・案件の同一性を **Salesforce の Consultation.Id / Matter.Id を唯一の脊椎ID** とし、Box・Gmail・MF 等は脊椎IDへ紐づく観測に留める。脊椎の付与・統合・分割は人間判断。— この理解でよいか。
- 現時点の仮説: SF=制御塔の前提と整合（D-001 仮説・未決）。
- 確認済証拠: Phase0 §7「主キー Consultation.Id / Matter.Id はそのまま流用が基本（D-001）」。Phase2 Box実査で案件は `_alo_dir.json`/`_alo_case_summary.md`（Derived）を持つがBox側に正本IDは無く、フォルダ名（頭文字_案件名_事件種別）で識別。
- 推奨回答案: SFの2IDを脊椎とする。Box/Gmail側はSF IDへの参照キーを持たせる。
- 選択肢: (A) SF 2IDを脊椎 / (B) Box file_id を脊椎 / (C) 独立した ALO 採番IDを新設し両者を束ねる。
- 回答で変わる設計: 全イベント・全文書の `matter_ref` の主キー定義、突合ロジック、SF↔Box連携方向。
- 具体例: Gmailトレースでは案件をスレッド単位で仮IDにしたが（Ledger `matter:trace1`）、本番では脊椎IDに解決する必要がある。
- 例外ケース: 受任前の相談段階（Matter未生成）／同一依頼者の複数案件／スポット相談で案件化しないもの。
- 回答者: 浅井（中森補佐）

## G-02. 文書の同一性の正本 [D-002 / Box-U5]

- **裁定**: 文書の同一性は **Box の file_id + version_id を正本** とし、共有リンクやファイル名は同一性判定に使わない。docx→pdf は別文書でなく「同一文書の確定/配布版」として版で連結。— この理解でよいか。
- 現時点の仮説: D-002（仮説・未決）。未解決論点は「共有リンクでなくID取得可否」。
- 確認済証拠: Phase2「docx⇔pdfペアが確定/PDF化境界の自然な検出点」「ひな形は複数版更新（期日請書v5・受任通知v8・明細照会書v9）」。同一性をファイル名に依存すると `振込先指定書` / `振込先指定書2` の連番派生で破綻する所見あり。
- 推奨回答案: file_id+version_id を正本。docx/pdf は `derived_from` で連結。
- 選択肢: (A) file_id+version_id / (B) ファイル名+更新日時 / (C) 共有リンクURL。
- 回答で変わる設計: 文書の重複排除、版管理、`document_refs.box_file_id/box_version_id` の必須化。

- 例外ケース: ~BROMIUM コンテナ内・`old` 退避版・Box外（Gmail添付のみ）の文書（G-15参照）。

- 現時点の仮説: D-003/D-010（仮説・未決）。未解決論点は「個人情報削除要求時の例外」。
- 確認済証拠: REQUEST §2 絶対原則「元メール・元録音・元文書を破壊・改変しない」。Ledger は全イベントを `invalidates_event_ids` 付き封筒で append-only 記録済（実装済みの設計萌芽）。

- 選択肢: (A) append-only＋無効化 / (B) 上書き訂正可 / (C) ハイブリッド（Raw不変・Canonicalは訂正可）。

## G-04. declared_state と observed_state の分離 [D-004 / P0:Q-005]

- **裁定**: 案件状態は **「人間が宣言した状態(declared)」と「データから観測された状態(observed)」を別物として併記** し、両者の乖離を異常として可視化する。正本はどちらか一方でなく、ゲート通過は declared を必須とする。— この理解でよいか。
- 現時点の仮説: D-004（仮説・未決）。
- 確認済証拠: Phase0「declared_state と observed_state を分離（D-004）」。Ledger は各遷移に `basis: observed|declared` を付与済（trace1 0103 は declared、0104 は observed）。
- 推奨回答案: 両併記。重大ゲートは declared 必須、observed は監視・先行検知に使う。
- 選択肢: (A) declared/observed 併記 / (B) observed 単独（自動）/ (C) declared 単独（手入力）。
- 回答で変わる設計: 状態機械の二層化、乖離アラート、KPI（状態と実態の乖離率）の定義。
- 具体例: 保険会社が「支払受付」と返信（observed=payment_acknowledged）だが入金消込は未確認（Ledger trace1 0106 コメント）。
- 例外ケース: observed が declared に先行する場合（入金観測済だが人間未承認）の扱い。
- 回答者: 浅井
