## 4. 設計への含意（v0.2 方針への接続）

1. **Raw / Canonical / Derived 分離は既に萌芽がある。** 案件フォルダ内に Raw（正本PDF/docx）と並んで機械生成 `_alo_case_summary.md` `_alo_dir.json`(Derived) が共存。設計では Derived 層を明示的に分離・再生成可能にする（Rawは不変、Derivedは派生）。
2. **状態はファイル名・フォルダ配置という「弱いシグナル」に埋め込まれている。** 送付/返送/押印前、進行中/既済、docx/pdf。これを制御塔(SF)側の構造化ステータスへ昇格させると、到達・受理・入金など Box 単独では △/× の状態が補完できる。
3. **docx⇔pdf ペアが「確定/PDF化」境界の自然な検出点。** 同名 docx と pdf の存在・更新時刻差で「編集中→確定発行」を機械判定できる（書式・マニュアル・契約書で共通パターン）。
4. **会計系（見積→請求→領収）と通信系（送/受）は分離管理され、台帳ファイル（請求書発行管理簿 v5、追跡番号記録表 v8）が縦串。** これらの台帳が SF 制御塔との接続点になる。Box 文書＝正本、台帳/SF＝状態という役割分担が観測事実と整合。
5. **命名規約が明文化済み（案件フォルダルール.txt、案件名特定CSV）。** 既存規約を尊重し、機械パースしやすい正規化（頭文字_案件名_事件種別、0-/1-/2-/4- 番号体系）を設計の所与とできる。
6. **`old` フォルダと `~BROMIUM` の扱いを設計で定義要。** old=廃版/重複退避（廃版の弱表現）、~BROMIUM=セキュア分離コンテナ痕跡。Raw不変原則との整合を要確認。

## 6. 実査の制約・遵守事項
