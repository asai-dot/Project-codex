## 0.5 Box↔SF 対応資産の選定（複数トライの比較・どれを渡すか）

Box↔Salesforce の対応には複数のトライがあり、**`alo_dir_index_accum.json`（=`alo-sf-box-index-v1`）は最終ではない**（214/332＝**64%・2026-03-30 の部分版**で古い）。AIフィットへ渡すべきは以下の**併用**：

| 資産 | 役割 | 規模 | 使いどころ |
|---|---|---|---|
| **`_alo_dir.json`**（alo-dir-map-v2 / DD-DIR-V2） | **フォルダ→SF ID の確定アンカー**（人手確定・最精度）＋サブ分類・通信方向 | 案件**423件**紐付け済 | 正解ラベルの**確定**、構造の取得 |
| **DD-BOXCATALOG-001 常設索引**（candidate・稼働中） | **最新・最広**の案件↔ファイル↔SF照合（毎日更新, SQLite） | **box_case_folders 495 / box_case_files 59,017 / SF照合 403案件 / file_with_case 55,135** | 網羅的な正解ラベル一括生成・ファイル探索 |
| DD-BMANIFEST-001 v0.2（accepted） | ファイルメタ feed（20列, `content_sha1`, `box_file_id`） | 3 LEALA×13 shard（メタのみ） | ファイル単位の差分・内容変化検出 |
| ~~`alo_sf_box_index_v1.json`~~ | 集約index初版 | 64%・古い | **使わない（superseded）** |

→ **精度の確定は `_alo_dir.json`、網羅の最新最広は DD-BOXCATALOG-001 の常設索引**。この2つを併用するのが最良。古い `alo_sf_box_index_v1`/`alo_dir_index_accum.json` は紹介しない。

`_alo_dir.json` は AIフィットにとって**最重要資産**。1フォルダにつき以下を持つ：
- `sf`（SF Record ID）/ `alo_matter_uri` / `case`（案件名）/ `case_root_box`
- `subs[]`: サブフォルダの **分類(`cat`)**（訴訟記録/書面/債権執行/通信_受/通信_送/会計/文書データ/その他）、**方向(`dir`: incoming/outgoing)**、**通信受↔送のペア(`peer`)**、**AI処理フラグ(`ai`: normal/deprioritize/exclude)**、サブ事件(0-0/0-1…)
- `case_summary_box_file_id` / スキャン状態 / `matter_status`

## 2. §F 当事者ロールの意味論（ベンダーが最も誤る・要伝達）

ALOコネクトは「誰からのメールか」を**案件の当事者構造に正しくマッピング**する必要がある。一般AIベンダーには非自明：

- **相手方・相手方代理人からのメールも「自分の案件」に紐付く**。差出人が相手方弁護士でも、それは当該matterの活動（`OpponentParty__c`126件 / 代理人）。
- **自所が当事者のケース**: 弁護士が**破産管財人 / 後見人 / 遺言執行者**のとき、ALO自身が手続主体（依頼者ではない）。「浅井」名でも内部混同しない。
- **共同受任・特約**（浅井＋武市特約 等）: 1matterに複数担当。
- **紹介元 ≠ 当事者**（法テラス/弁護士会/遺言相続センター/夜間相談）: 入口であって当事者ではない。
- **協業先 ≠ 依頼者**（会計士/司法書士/FA）: M&A等の協業者。
- **裁判所からの送達**: 当事者でなく court（`JurisdictionCourt__c`160件、事件番号が最強キー）。
- SF対応: `RelatedContact__c`(1,744) 関係者 / `OpponentParty__c`(126) 相手方 / `matter_party.party_role`。

## 3. §G 難所カタログ / ゴールドセット（最優先で作るべき抜け漏れ）

AIフィットが resolver を**作り・検証する**には、ラベル付きの実例集が要る。これは**ALOしか作れないドメイン資産**。

各エントリ案（匿名化可）:
```
signal（差出人/件名/番号/本文断片）→ 正解 sf_case_id（or「非案件」）
理由（どのルールで当たるか: 9パターン/旧姓/相手方/事件番号/…）
難所タグ（旧姓立件 / 同姓異人 / 複数案件 / 相手方代理人 / 協業先 / 中黒 / SF未起票 / 紹介元 …）
confidence想定（auto/review/unmatched）
```
- 既に手元にある実証例（→ そのまま種にできる）: 旧姓立件(婚氏↔旧姓)、同姓異人、中黒表記ゆれ、利益相反の消去法担当、協業先=非案件、複数案件(法人＋代表者個人)、関連交渉=受任サブ。
- 用途: AIフィットの**特徴量設計 / confidence較正 / 回帰テスト**。`eval_set` テーブルにそのまま載る。

## 4. §H AIフィットが踏みやすい罠（負の知識・1枚化推奨）

| 罠 | 正 |
|---|---|
| @メンション=担当 | 利益相反確認の@は**非担当**（消去法）。単独で担当確定しない |
| 同姓=同一人 | 別人は別party。evidence揃いでのみ併合 |
| メール件数が多い=重要案件 | 研修・対外活動は件数多くても**非案件** |
| 紹介元=当事者 | 入口に過ぎない |
| 顧問先の定例連絡=個別案件 | 顧問は「土壌」。個別受任と分離 |
| 旧社名/婚氏で別物 | 旧社名・旧姓は別名（要名寄せ、ただし自動確定しない） |
| 中黒/全半角を潰せば安全 | 潰しすぎると**別法人を誤併合**。正規化キーは候補用、確定キーにしない |
| Boxに背骨IDが無い | **ある**（`_alo_dir.json` の `sf`） |

## 5. §J データ品質ランドマイン（先に警告すべき地雷）

- **ラベル破損 41件**（`0015h…`等だけ表示）→ TASK-A対象。
- **SF未起票 ≈ 30%**（Pattern E。Box実体ありSF無し）→ 単純JOINで欠落。
- **重複起票**（同名で受任＋関連交渉が両方＝龍村売掛金回収 等）。
- **漢字統一未**（河野 弘/博史/宏 等の同一人）。
- **ghost object・遺物**（`leala__Procedure__c`不存在＝domain別*Process__c / `leala__Matter__c`は1件遺物）→ 3月資料の幽霊objectを実在扱いしない。**実SF describe / SCHEMA-REALITY 優先**。
- 動的DB実体: Supabase `dynamic` schema は**実在14テーブル**（parties/cases/comms/documents/…/unconfirmed_links/…）。`reconciliation_issue` 等は**同名で実在しない**（設計と実体の差に注意）。

## 7. 守秘の前提（AIフィットへ渡す際）

- AIフィットは外部ベンダー。**実依頼者情報を渡すには秘密保持の枠組み＋匿名化/事件ID化**が前提（GPT監査レーンと同基準）。難所カタログ等は**匿名化版**で渡し、実データ接続は契約・権限境界の中で。
- ALOコネクトの書込先（どのテーブルにAIフィットが書き、どこは事務所が書くか）は §A の照会で確定する。
