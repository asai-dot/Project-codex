## このパイロットで判明した重要点
1. **骨格はDBに既存**: `biblio.bib_toc` に章節〜小見出しが**頁付き4階層**で入っている（目次OCR不要）。
   - 注: `page` は **スキャンPDFのページ順（表紙=0）**。→ PDFの1-based頁 = `bib_toc.page + 1`。
2. **14式は独立した目次ノードではない**: 本書の書式（モデル契約書・合意書・解除通知例）は
   **ケーススタディ本文に埋め込み**で、TOCに「書式」見出しが立っていない。
   → 式の所在は「bencomの書式リスト or 本文のポイントOCRでの書式検出」で取る必要がある。
3. **画像取得はMac側必須**: リモート環境の Box `get_preview_page` は空(0byte)を返す（40MB/小ファイルとも）。
   → `tools/pointocr_pilot.py` は Box の **PNG representation** を直接取得して vision に渡す設計。

## ⚠ 運用上の境界：Box変換のサイズ上限（重要）
MCPプレビューOCRは Box の PNG変換に依存。**実測の可否**:
| ファイル | サイズ | 変換 |
|---|--:|---|
| 契約解消 01_全体 | 40MB | ✅ |
| 業務委託 01_全体 | 220MB | ✅（初回0byte→温め直しで成功） |
| 会社法務書式集 01_全体 | 365MB | ✅（pending待ち後に成功） |
| 契約書作成の実務と書式 01_全体 | **642MB** | ❌ 404 conversion_failed |

→ **変換上限は約365MB（可）〜642MB（不可）の間**。365MB級までは「pendingを1分ほど待って再取得」すれば変換される。
それを超える600dpiフル本は、この環境内OCR不可（Mac側レンダリングへ）。

**会社法務書式集（本丸級・238式）は既存の旧版スキャン(365MB)からOCR可能と実証**（`out_kaishahoumu_shoshiki1-8.json`＝発起人決定書を【記入例】付きで構造化）。
ただし自炊は旧版・bib_tocは第3版のみで頁不一致のため、**正規の生材料化は第3版の再自炊が本筋**（共通書式は旧版スキャンで先行着手可）。
本丸の再自炊では、**OCR用に「変換可能サイズ(目安≤約250MB)」へ収める**運用が要る（§ 本丸再自炊指示 参照）。
それを超える既存大型スキャンは Mac側レンダリング（`pdftoppm`→画像→OCR）に回す。
