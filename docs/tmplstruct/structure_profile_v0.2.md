- **一行**: 書式は「読む文献(outline)」でなく「再生成できる型」。**復元単位＝アーキタイプ（定型span＋差込slot＋繰返group）**。formType/category は型に乗る**ラベル**にすぎない。

## 0. 30件実証で確定した事実（検収根拠）
`structure_clauses_ocr` は `第1条/第1章` 等の**番号マーカーのみ**で、**条見出しキャプション（例「(売買の目的物)」）は欠落**、かつ**抜け**あり（14920: 第3→第5条で第4条落ち / 9332: 第6→第10条で7–9落ち）。
→ **OCRは構造の「骨格・形状」は拾えるが「文面」は壊す**（甲→と、売買→完買、表は崩壊）。
| カテゴリ | n_clauses_ocr の実態 | 形状の含意 |
|---|---|---|
| 契約8 | 5〜15（ただし 発起人同意書=0, 英文NDA=0） | 大半=条文型。例外2件は別形状（表/英文） |
| その他8 | 定款43・職務発明規程37 ／ 他6件=0 | **二極化**：条文型 と 記載例型 が同居 |
| 申立5・通知4・登記4 | **全件0、1ページ** | 条が無い＝**slot型**。監査の slot 仮説を実証 |
| 公正証書1 | 4 | 条文型に近い |

## 1. 構造アーキタイプ（復元単位＝5形状）
各書式は **title/category ではなく構造シグナルで** 次の archetype に分類する。

| archetype | 検出シグナル（OCRで判定可） | 代表 | 復元 source |
|---|---|---|---|
| **A. clause_contract** | `第N条`/`第N章` が複数。前文「甲/乙…締結する」。括弧見出し+項。 | 契約, 定款, 各種規程, 公正証書遺言 | 高価値→**.docx必須**（条見出し/文面の忠実復元）。OCRは分類のみ |
| **B. header_petition** | 宛先`◯◯裁判所 御中`/長＋申立人ヘッダ(〒住所/氏名)＋`申立の趣旨/原因/目的` ラベル節 | 申立書, 各種申請書 | OCRで slot 復元可。.docxは任意 |
| **C. keyvalue_filing** | `ラベル：値` の箇条（`1.商号…/1.登記すべき事項 別紙のとおり`）＋申請人＋官公署宛 | 登記申請書, 届出 | OCRで slot 復元可 |
| **D. prose_notice** | 条なしの自由文本体＋差込slot（金額/利率/期日/当事者）＋差出/宛先 | 通知書, 催告書, 内容証明 | OCRで概ね可（誤字注意） |
| **E. table_or_record** | 表セル主体／短い記載例スニペット | 議事録, 記載例, 公告, チェックリスト, 同意書(表) | 表は**.docx推奨**（OCR表崩壊）。記載例はOCR可 |

> 6カテゴリ→5archetype は **多対多**（例: `その他`→A(定款/規程)とE(記載例/議事録) に分かれる、`契約`の発起人同意書は実はE）。**だから title分類だけでは復元単位を割れない＝形状分類器が要る**（v0.1 の盲点）。

## 2. 再復元スキーマ（archetype 共通の器）
```jsonc
restorable_profile = {
  template_id, archetype,                 // A–E
  formType_label,                         // 表示用ラベル（後述の入口分類）
  fixed_spans: [ {text, role} ],          // 定型文（前文/結語/ラベル/官署宛 等）
  slots: [ {name, kind, pattern, repeat?} ],
  repeat_groups: [ {name, unit, of} ],    // 条/項/号、署名当事者、明細行
  signature_block?: { date_slot, parties:[{role, name_slot, address_slot, seal?}] },
  source_fidelity: "ocr" | "docx_required" // 忠実復元に docx が要るか
}
```
- **slot.kind 語彙（実物由来）**: `party(甲/乙+定義)` `person_name` `address(〒)` `date(和暦)` `money(金◯◯円)` `rate(年◯%/◯分)` `authority(◯◯裁判所/法務局 御中)` `enum_items` `table_cell` `free_blank(◯◯/〇〇/下線)`。
- OCRは空欄を `◯◯ / 〇〇 / 下線` で表現 → これを slot 境界の検出器にできる（A〜Dで有効）。

## 3. formType（入口）の建て直し — 「その他45→65%」を割る
- **入口は2層**: ① `category_id_lib`(1〜9＝法律ライブラリ純正タクソノミ。全レコード保持) を第一候補に採用（推測より強い）。② それを `archetype`(A–E) で直交クロス。
- 「その他」は**捨て場でなく未分類フラグ**として扱い、archetype が付けば実質解消（定款=A, 議事録=E…）。
- 効果見込み: `その他` の多くは A(規程/定款) か E(記載例/議事録/公告) に再配分され、真の未分類だけが残る。

## 4. OCR と .docx の役割分担（クォータ最適化）
- **OCR(無料・既存3,806全件)** = ①archetype分類 ②slot/領域検出 ③全文検索。**1%サンプルで実行可能性を実証**。
- **.docx(月30枠・legal-library.jp/r/{book}?page=N, filetype=Word)** = **A(契約/定款/規程)と E(表) の高価値・高再利用だけ忠実復元**。OCRは条見出し/文面を落とすため復元には不可欠。
- 投資順: 契約559 のうち利用頻度上位 → 月30で段階。その他2,476内のA(規程/定款)を archetype 分類で発掘して優先度付け。

## 5. 検収結果（ワーカーの4観点への回答）
1. **分類規則の妥当性**: 妥当だが**title正規表現は粗い**。訴状を「申立」に入れる/JV を「契約」は許容。ただし `発起人同意書→契約` は実体E（表）で誤り、`定款/規程/議事録→その他` は実体A/E。→ **PASS_WITH_NOTES。次版は category_id_lib＋archetypeに移行**。
2. **配分8/5/4/4/1/8**: 検証目的としては妥当（全archetypeを踏めた）。ただし母数は 契約559/その他2,476。**事務所利用比率での再重み付けは別途**（owner入力待ち）。
3. **structure_clauses_ocr 品質**: **不十分**。番号のみ・キャプション欠落・抜けあり。**分類には足りるが忠実復元には docx 必須**（§0,§4）。
4. **再復元方式の合意可否**: 本サンプルで **§1–§4 の archetype 方式が引けた**。→ **3,806横展開＝OCRで archetype分類器を回す（無料）/ docx は高価値Aのみ**で合意提案。
